# PMP Exam Master Brief: Sequence Activities

---

## 1. Topic Snapshot — Plain English

Sequence Activities is the process of identifying and documenting relationships among project activities to create a logical schedule network. It takes the activity list, activity attributes, and milestone list from Define Activities and produces a project schedule network diagram with any activity sequence updates. The core technique is the Precedence Diagramming Method (PDM), which uses nodes for activities and arrows for logical relationships (Finish-to-Start, Finish-to-Finish, Start-to-Start, Start-to-Finish). Beyond simple dependencies, Sequence Activities requires the PM to document dependency types (mandatory, discretionary, external, internal), apply leads and lags accurately, and ensure every activity has at least one predecessor and one successor except for the first and last activities. This process feeds directly into Estimate Activity Durations and is the structural backbone of the entire schedule. Approximately 10–12% of the PMP exam covers Schedule Management; sequencing questions test dependency type discrimination, lead/lag application, and network logic integrity. **In one sentence:** Sequence Activities builds the logical skeleton of the project schedule by defining how every activity relates to every other activity through typed, quantified dependencies.

## 2. The PMI Mindset on This Topic

### Predictive vs Agile vs Hybrid Lens

| Lens | What it looks like |
|---|---|
| Predictive | Full network diagram built upfront; all dependencies documented before scheduling; leads and lags explicitly justified; PDM used throughout |
| Agile | Dependencies managed at story/task level within sprint planning; network diagrams not typically used; dependency impediments surfaced in daily standups and resolved by scrum master |
| Hybrid | Regulatory/compliance activity chains sequenced predictively with mandatory dependencies; agile feature development tasks sequenced sprint by sprint; integration points between predictive and agile tracks have explicitly documented interface activities |

### Top 5 PMI-isms on This Topic

1. **Discretionary dependencies are the PM's choice and can be changed.** Real-world PMs treat all dependencies as fixed constraints; PMI distinguishes mandatory dependencies (inherent in the work) from discretionary ones (PM preference or best practice) that can be altered when schedule needs to be compressed.
2. **Lag is not free time — it is a waiting period that must be justified.** Practitioners insert lags to buffer against uncertainty; PMI treats unexplained lags as schedule padding (disguised reserve) that must be explicitly justified with a reason tied to the actual work.
3. **Leads accelerate the successor, not just overlap.** PMs often describe overlapping activities as "working in parallel"; PMI's lead concept is a specific quantity (negative lag) applied to a dependency relationship that compresses schedule by allowing a successor to start before its predecessor finishes.
4. **Mandatory dependencies are not negotiable, even under schedule pressure.** When sponsors or stakeholders push for schedule compression, real-world PMs are tempted to remove mandatory dependencies; PMI treats mandatory dependencies as non-negotiable and routes compression through leads, lags, crashing, or fast-tracking discretionary dependencies.
5. **Every activity needs a predecessor and successor (except the first and last).** Network diagrams with dangling activities (no predecessor or no successor except at network ends) represent logic errors; PMI expects the network to be complete and closed.

### Ethics Angle

- **Responsibility:** Documenting mandatory dependencies honestly even when doing so exposes an uncomfortably long schedule; manufacturing discretionary dependencies to protect schedule float is misrepresentation.
- **Honesty:** Presenting lags as technical requirements when they are actually buffer time to hide uncertainty violates honesty norms.
- **Fairness:** In a capital markets project (BSEC system integration), sequencing that systematically delays vendor activities to protect internal milestones at the vendor's expense is unfair and must be documented and negotiated transparently.

## 4. Key Terms, Formulas & Acronyms

| Term / Formula | Definition / Calculation | Exam Trap to Avoid |
|---|---|---|
| Precedence Diagramming Method (PDM) | Network diagramming technique using nodes (boxes) for activities and arrows for dependencies; also called Activity on Node (AON) | Trap: confusing PDM/AON with Arrow Diagramming Method (ADM/AOA); PDM is current standard; ADM is legacy; PDM nodes = activities, ADM arrows = activities |
| Finish-to-Start (FS) | Successor cannot START until predecessor FINISHES; most common dependency | Trap: applying FS when SS or FF is more accurate, creating unnecessary schedule extension |
| Finish-to-Finish (FF) | Successor cannot FINISH until predecessor FINISHES; used when completion is dependent | Trap: confusing FF with FS; FF allows successor to start anytime but constrains finish |
| Start-to-Start (SS) | Successor cannot START until predecessor STARTS; used for parallel work with shared start | Trap: using SS without a lag when there is a realistic delay between starts |
| Start-to-Finish (SF) | Successor cannot FINISH until predecessor STARTS; rare; used in shift handovers | Trap: applying SF in standard sequences; it is almost never the right answer except for specific handover scenarios |
| Lead | Amount of time a successor activity can be advanced relative to its predecessor; expressed as negative lag | Trap: treating lead as optional or cosmetic; lead compresses schedule and must be technically justified |
| Lag | Waiting time between predecessor and successor; adds time to the network | Trap: inserting lag as disguised buffer reserve without documenting the reason; PMI requires lag to represent real waiting time |
| Mandatory Dependency | Dependency inherent in the nature of the work; legally or physically required; also called "hard logic" | Trap: treating mandatory dependencies as compressible; they cannot be changed without changing the nature of the work |
| Discretionary Dependency | Dependency based on best practice or PM preference; can be modified to compress schedule; also called "soft logic" | Trap: treating all dependencies as mandatory; discretionary dependencies can be reversed or modified in fast-tracking |
| External Dependency | Dependency involving a relationship between project activities and non-project activities; outside PM control | Trap: confusing external dependency with mandatory; external dependencies may be discretionary (e.g., waiting for a non-mandatory vendor report) |
| Internal Dependency | Dependency between project activities within the PM's control | Trap: treating internal mandatory dependencies as external to deflect accountability; internal mandatories are still the PM's responsibility to manage |
| Schedule Network Diagram | Visual representation of logical relationships among project schedule activities; produced by Sequence Activities | Trap: confusing network diagram with Gantt chart; Gantt shows time; network diagram shows logic |
| Dangling Activity | Activity with no predecessor or no successor (except network start/end); represents a network logic error | Trap: allowing dangling activities; every interior node must have both a predecessor and a successor |
| Fast-Tracking | Schedule compression technique: performing discretionary-dependency-linked activities in parallel that were originally sequential | Trap: fast-tracking mandatory dependencies; only discretionary dependencies can be fast-tracked |

## 5. PM Behavior Cheat Sheet — FIRST / NEXT / BEST / EXCEPT

| Stem Signal | PM Should FIRST Do | Because |
|---|---|---|
| "Schedule shows an activity with no predecessor" | Identify the logic error and assign an appropriate predecessor | Dangling activities corrupt critical path calculation; all interior activities must be fully connected |
| "Sponsor wants to compress schedule; dependencies seem flexible" | Identify discretionary dependencies and evaluate fast-tracking opportunities before agreeing to any compression | Mandatory dependencies cannot be changed; only discretionary dependencies are candidates for fast-tracking |
| "Vendor activity cannot start until regulatory approval from Bangladesh Bank is received" | Document this as an external mandatory dependency with the specific regulatory trigger | External mandatory dependencies must be explicitly documented with the triggering event so schedule risk can be tracked |
| "Team wants to overlap testing and development to save time" | Evaluate whether testing can start before development finishes; if yes, apply a lead to an FS or SS relationship | Overlapping activities are modeled with leads, not by removing the dependency relationship |
| "Lag has been inserted between two activities with no documented reason" | Investigate and document the technical justification for the lag or remove it | Undocumented lags represent hidden schedule padding; PMI requires all lags to reflect real waiting time |
| "Network diagram shows an activity connecting to the end without going through QA review" | Add QA review as a required predecessor to the final milestone | Network completeness requires all mandatory activities to be in the logic chain; bypassing QA in the network does not eliminate the work |
| "Team argues that a discretionary dependency is mandatory" | Review the technical basis; if the dependency can be safely removed without violating law, contract, or physical reality, classify it as discretionary | Dependency classification affects compression options; misclassifying discretionary as mandatory limits schedule flexibility |
| "PM is adding lags to multiple activity pairs to protect the schedule" | Review whether lags represent real waiting time; if they are protective buffer, move the time to schedule reserves | Disguised buffer lags mislead stakeholders about schedule logic; PMI reserves belong in contingency reserve, not embedded in activity lags |

**Topic-specific escalation edge cases:**

| Stem Signal | PM Should FIRST Do | Because |
|---|---|---|
| "Functional manager insists on removing a mandatory dependency to meet the sponsor's date" | Decline to remove the mandatory dependency; document the constraint, show its schedule impact, and explore discretionary dependency modifications instead | Removing mandatory dependencies falsifies the schedule and creates execution risk; PMI treats this as a schedule integrity issue |
| "External regulatory approval (BSEC) is on the critical path and will delay the project" | Inform stakeholders of the dependency and its schedule impact; explore whether any predecessor activities can be compressed to recover time upstream | External mandatory dependencies are non-negotiable; the response is transparent communication and upstream compression, not removal |

## 6. Visual Memory Aid

```
DEPENDENCY TYPES — DECISION TREE

Is the dependency required by law, contract, or physical reality?
  YES --> MANDATORY (hard logic) -- Cannot be removed; can compress elsewhere
  NO  --> DISCRETIONARY (soft logic) -- Can be modified for fast-tracking

Is the dependency party external to the project?
  YES --> EXTERNAL (e.g., regulatory approval, vendor delivery)
  NO  --> INTERNAL (within project team/sponsor control)

Combined: Mandatory + External = Hardest constraint
          Mandatory + Internal = Inflexible but PM owns it
          Discretionary + External = Negotiate
          Discretionary + Internal = Fast-track candidate

PDM RELATIONSHIP TYPES:
  FS: A ----finish----> B start       (most common)
  SS: A --start-------> B start       (parallel start)
  FF: A ----finish----> B finish      (parallel finish)
  SF: A --start-------> B finish      (rare: shift handover)

LEAD vs LAG:
  Lead (-lag): B starts BEFORE A finishes   --> schedule compression
  Lag  (+lag): B waits AFTER A finishes     --> required waiting time
```

**Mnemonic: MDIE-FS** — Mandatory, Discretionary, Internal, External — and the default relationship is Finish-to-Start

**Memory hook:** "My Dog Is Easy — Finish Strong" — classify every dependency (My=Mandatory, Dog=Discretionary, Is=Internal, Easy=External) then apply the default Finish-to-Start relationship unless the work demands otherwise.

## 7. Tailoring Considerations

### Predictive Project
- Complete network diagram required before critical path can be calculated
- All dependency types explicitly documented; mandatory dependencies protected from compression pressure
- Leads and lags recorded with technical justification in activity attributes
- Network must be reviewed for dangling activities before schedule development
- Change control governs any dependency changes after baseline

### Agile / Hybrid
- Sprint-level dependency management replaces formal network diagrams for agile components
- Dependencies between agile tasks surface in daily standup and backlog refinement
- Hybrid projects: predictive activity chains for regulatory deliverables use formal PDM; agile sprint tasks managed visually on task boards
- Integration points between predictive and agile tracks (e.g., a regulatory review milestone that blocks an agile sprint from releasing) are explicitly sequenced and owned

### Industry / Regulatory Factors
- **Bangladesh Bank IT guidelines:** Parallel run activities for core banking changes have mandatory dependencies on UAT completion; these are hard logic constraints that cannot be removed under schedule pressure
- **BSEC reporting systems:** External mandatory dependency on BSEC review and no-objection certificate before system go-live; PM must document this as an external constraint with buffer for regulatory processing time
- **Capital markets trading systems:** DSE/CDBL interface certification is an external mandatory dependency; schedule must reflect certification lead time realistically, not optimistically
- **Regulated financial environments:** Auditors review network diagrams for logic integrity; undocumented lags or missing mandatory predecessors create audit findings

## 8. Common Wrong Answers & Why PMI Rejects Them

| Tempting Answer | Why Experienced PMs Pick It | Why PMI Marks It Wrong |
|---|---|---|
| "Remove the dependency between testing and deployment to compress the schedule" | Practical PMs often skip formal testing sign-off in crunch; seems like a reasonable trade-off | If the testing→deployment dependency is mandatory, it cannot be removed; removing it falsifies the schedule and creates execution risk |
| "Add lag between all major milestones to protect the schedule" | Experienced PMs know things take longer than planned; protective lags seem prudent | Undocumented protective lags are disguised contingency reserve; PMI requires lags to represent real technical waiting time, not buffer |
| "Use a Start-to-Start relationship so both activities run in parallel" | SS looks like an efficient parallelism tool | SS constrains when the successor can start but does not model overlapping completion correctly; if the intent is overlap, FS + lead is often more accurate |
| "All dependencies on the critical path are mandatory" | Critical path activities feel most important and inflexible | Criticality and mandatory status are independent; a critical path activity may have a discretionary dependency that could be fast-tracked |
| "Fast-track by removing mandatory dependencies to save time" | Fast-tracking sounds like removing constraints; experienced PMs conflate dependency removal with fast-tracking | Fast-tracking applies only to discretionary dependencies; removing mandatory dependencies creates physical or legal impossibility |
| "External dependencies are not the PM's responsibility to manage" | Vendor and regulatory delays are outside the PM's control; feels logical to leave to the vendor | External dependencies must still be documented, tracked, and communicated; the PM owns the schedule impact even if not the activity itself |
| "Lags represent buffer time built into the schedule for risk" | Adding buffer is standard practice; building it into lags seems like good planning | PMI requires lags to represent real technical waiting (e.g., concrete curing time); risk buffer belongs in schedule reserves, not embedded in lags |
| "Network diagrams and Gantt charts show the same information" | Both are schedule tools; PM routinely uses them interchangeably | Network diagrams show logic (dependencies); Gantt charts show time; critical path calculation requires the network, not the Gantt |

## 11. Exam-Day Trigger Phrases

| Trigger Phrase in Stem | Your Reflex Answer Pattern |
|---|---|
| "Activity cannot start until another activity is complete" | Finish-to-Start dependency — the most common relationship |
| "Physically impossible to do B before A" | Mandatory dependency — hard logic, cannot be removed |
| "Best practice dictates this sequence" | Discretionary dependency — soft logic, can be changed for compression |
| "Regulatory approval required before proceeding" | External mandatory dependency — document with specific trigger, track as schedule risk |
| "Testing can begin before all development is complete" | Apply a lead to the FS or SS relationship — do not remove the dependency |
| "Schedule must be compressed; what dependencies can be changed?" | Identify discretionary dependencies as fast-track candidates; mandatory dependencies are off-limits |
| "Lags have no documented reason" | Investigate — undocumented lags are likely hidden buffer; justify or remove |
| "Activity has no successor" | Logic error — dangling activity; assign a successor before calculating critical path |
| "Two activities must finish together" | Finish-to-Finish relationship |
| "Two activities must start at the same time" | Start-to-Start relationship (with or without lag) |
| "Vendor activity is delaying project milestone" | External dependency impacting schedule — communicate impact, explore upstream compression |
| "PM adds buffer to every activity link" | PMI red flag — embedded buffer belongs in reserves, not lags |
| "Fast-tracking is proposed but activities have hard logic" | Cannot fast-track mandatory dependencies — propose crashing instead |
| "Network shows parallel tracks merging at a milestone" | Ensure all parallel tracks are properly connected to the milestone; check for danglers |

## 9. 10 Hardest Scenario-Based Questions

**Q1 — Ethics Dilemma (Predictive)**

A project manager is finalizing the schedule network diagram for a Bangladesh Bank core banking system upgrade. The project sponsor, facing a board presentation deadline, asks the PM to remove the mandatory UAT-completion-to-parallel-run-start dependency from the schedule to show a four-week schedule reduction. The sponsor argues that in practice "the teams always overlap these anyway." The PM knows Bangladesh Bank IT guidelines explicitly require UAT sign-off before parallel run begins. The PM should:

A. Remove the dependency as requested; the sponsor has authority over the schedule and the UAT/parallel-run overlap can be managed operationally without affecting the baseline.
B. Reclassify the dependency from mandatory to discretionary so the sponsor can see the schedule with fast-tracking applied, presenting a more optimistic scenario.
C. Decline to remove the mandatory dependency; document the regulatory basis for the dependency, show the sponsor the realistic schedule, and present alternative compression options affecting discretionary dependencies.
D. Escalate to the PMO and the Bangladesh Bank relationship manager before responding to the sponsor's request.

---

**Q2 — Agile Ceremony / Engagement**

An agile development team building a capital markets trading dashboard is conducting sprint planning. Three user stories have been selected for the sprint. During task decomposition, the team identifies that Story 2 (trade summary report) has a dependency on Story 1's data model being finalized, but Story 1 is not yet complete. The dependency is known but the product owner says both stories should be in this sprint. How should the team handle this dependency?

A. Remove Story 2 from the sprint backlog; dependencies between stories in the same sprint are not manageable.
B. Acknowledge the dependency in the sprint plan, sequence Story 1 tasks before Story 2 tasks, and flag the dependency as a potential impediment for the scrum master to monitor.
C. Assign the same developer to both stories to eliminate the dependency through knowledge continuity.
D. Document the dependency in the schedule network diagram and submit a change request to resequence the sprint.

**Q3 — FIRST Action, Conflict (Hybrid)**

A hybrid project manager is sequencing activities for a BSEC regulatory reporting module. The development team lead wants to sequence the BSEC data validation activity after the user interface development is complete (FS relationship). A business analyst argues that data validation and UI development can run simultaneously since they use different datasets. Both have technical arguments. The project manager must decide FIRST. What should the PM do?

A. Accept the development team lead's FS sequence; the team lead has technical authority over sequencing decisions.
B. Accept the business analyst's parallel approach; running activities simultaneously saves schedule time.
C. Facilitate a technical review with both parties to determine whether the dependency is mandatory (work product required) or discretionary (process preference); then document the appropriate relationship.
D. Escalate the sequencing disagreement to the project sponsor for a final decision.

**Q4 — Two-Correct, Pick-Best (Predictive)**

A project manager reviewing a schedule network diagram for a capital markets system implementation discovers two issues: (1) a lag of 10 business days has been inserted between code freeze and testing start with no documented reason; and (2) a dependency between system integration testing and the BSEC certification review is labeled as discretionary. Which action BEST reflects PMI guidance?

A. Remove the undocumented lag and change the BSEC certification dependency to mandatory, then document both changes.
B. Leave the lag in place as a schedule buffer and reclassify the BSEC certification dependency to external mandatory since it involves a regulatory body outside the project.
C. Remove the undocumented lag (or document its technical basis) and reclassify the BSEC certification dependency as external mandatory, then document both changes.
D. Escalate both issues to the schedule analyst for resolution before making any changes.

**Q7 — Ethics / Honesty Hardline**

A senior project manager working on a BSEC trading system implementation has been asked by the project director to label all critical path dependencies as "mandatory" in the schedule network diagram. The director explains this is to "protect them from fast-tracking requests." The PM knows that several dependencies on the critical path are actually discretionary best practices, not mandatory requirements. The PM should:

**Q9 — Escalation Judgment (Hybrid)**

A project manager is three weeks into executing a hybrid project. The predictive compliance track has a formally approved network diagram. The PM discovers that a junior scheduler added a discretionary dependency between two critical compliance activities and, without a change request, relabeled it as mandatory to prevent the sponsor from fast-tracking it. The activities in question genuinely have a discretionary relationship. The PM is now under pressure from the sponsor to fast-track those activities. What should the PM do?

A. Support the scheduler's relabeling decision; the intent was to protect schedule integrity and the outcome aligns with PMI's preference for conservative scheduling.

## 10. Answer Key with Full Justification

**Q1 — Correct Answer: C**

**PMI Justification:** PMI Code of Ethics — Honesty and Responsibility; Bangladesh Bank IT regulatory requirements constitute a mandatory dependency. Removing a regulatory mandatory dependency falsifies the schedule and exposes the organization to regulatory non-compliance. PMI requires the PM to present an accurate schedule regardless of sponsor pressure. Option A fails because sponsor authority does not extend to falsifying mandatory dependencies, especially regulatory ones. Option B fails because reclassifying a mandatory regulatory dependency as discretionary to enable compression is intentional misrepresentation — it changes the label to achieve a deceptive result. Option D fails because the PM does not need PMO authorization to uphold schedule accuracy; the correct action is to explain and decline, not to delay.

**Mindset Lesson:** Mandatory regulatory dependencies are facts about the work, not schedule preferences — the PM's job is to present them accurately, not to find ways to make them disappear.

---

**Q2 — Correct Answer: B**

**PMI Justification:** Agile Practice Guide — sprint-level dependency management. In agile, dependencies between sprint items are managed within the sprint through transparent sequencing and impediment tracking, not by removing stories or creating formal network diagrams. The scrum master role includes impediment removal, which makes flagging the dependency for scrum master monitoring the correct agile response. Option A fails because removing a story solely because of a manageable dependency eliminates value unnecessarily. Option C fails because assigning one developer to both stories is a resource allocation workaround that does not resolve the dependency. Option D fails because agile teams do not use formal PDM network diagrams or submit change requests for sprint-level task sequencing.

**Mindset Lesson:** In agile, manage sprint dependencies transparently through sequencing and impediment tracking — do not eliminate stories to avoid dependencies.

---

**Q3 — Correct Answer: C**

**PMI Justification:** PMBOK — Define Activities and Sequence Activities; dependency classification requires technical assessment, not unilateral decision by either party. The PM's role is to facilitate the determination of dependency type. If the dependency is mandatory (validation requires UI data output), it must be documented as FS or FF. If it is discretionary (teams can work independently with interim data), a lead or SS relationship is appropriate. Option A fails because the PM cannot abdicate dependency classification to a team lead without technical assessment. Option B fails because accepting parallel sequencing without confirming the technical basis could create rework if the dependency is actually mandatory. Option D fails because this is not an ambiguous authority issue requiring sponsor resolution; it is a technical dependency question requiring facilitated expert input.

**Mindset Lesson:** Dependency classification is a technical determination requiring expert input — the PM facilitates the assessment, not the decision by default or authority.

---

**Q4 — Correct Answer: C**

**PMI Justification:** PMBOK — lags must represent real technical waiting time; regulatory body dependencies are external mandatory. The undocumented 10-day lag either reflects real waiting time (in which case it needs documentation) or is hidden buffer (in which case it should be removed or moved to reserves). The BSEC certification dependency involves an external regulatory body, making it external mandatory — not discretionary. Option A is partially correct but incomplete; "remove" the lag without investigating assumes it is buffer when it might be legitimate; the correct response is to document the basis or remove it, not blindly remove. Option B fails by retaining undocumented lag (schedule padding) and misclassifying BSEC (external mandatory, not discretionary). Option D fails because delegating both issues to a schedule analyst avoids the PM's accountability.

**Mindset Lesson:** Lags without documented reasons are presumed buffer; regulatory body dependencies are external mandatory — both classifications require immediate correction.

---

**Q5 — Correct Answer: A**

**PMI Justification:** Agile Practice Guide — sprint-level dependency management sequence. The logical order is: first identify dependencies, then sequence tasks to honor them, then flag what cannot be resolved internally, then the scrum master addresses impediments, and finally the team executes. You cannot sequence before identifying, cannot flag before sequencing, cannot resolve before flagging, and cannot execute before resolution. Options B, C, and D all violate this logical order in different ways.

**Mindset Lesson:** In agile, dependency management is a sequential process within sprint planning: identify first, sequence second, flag third, resolve fourth, execute last.

---

**Q6 — Correct Answer: C**

**PMI Justification:** PMBOK — PDM relationship types. PDM defines four relationship types: FS, FF, SS, and SF. "Activity-to-Milestone" is not a PDM relationship type — it describes a connectivity pattern, not a logical relationship. Milestones are activities with zero duration and participate in FS and other relationships like any other activity; they do not constitute a separate relationship type. Options A, B, and D are all valid PDM dependency types.

**Mindset Lesson:** PDM has exactly four relationship types: FS, FF, SS, SF — everything else is a description of the network, not a relationship type.

---

**Q7 — Correct Answer: B**

**PMI Justification:** PMI Code of Ethics — Honesty; PMBOK — dependency classification accuracy. Misclassifying discretionary dependencies as mandatory deliberately misrepresents the schedule and removes legitimate compression options from future planning. The PM's obligation is to document dependencies accurately and explain the professional basis for doing so, even when directed otherwise by a senior colleague. Option A fails because following instructions to misrepresent a plan document is a direct honesty violation regardless of the director's seniority or intent. Option C fails because maintaining a private accurate record while presenting false documentation is still deceptive — the official plan is what stakeholders rely on. Option D fails because while the request is improper, immediate ethics hotline reporting before attempting to resolve professionally is disproportionate as a first response.

**Mindset Lesson:** Intentional misclassification of dependencies — even with protective intent — falsifies the schedule; accuracy is non-negotiable regardless of who issues the instruction.

---

**Q8 — Correct Answer: B**

**PMI Justification:** PMBOK 7 — tailored communication; hybrid project stakeholder engagement. The correct approach for a hybrid project is to provide appropriate artifacts for each delivery track rather than forcing one format across both. A network diagram for the predictive track and a roadmap/dependency summary for the agile track, combined with a facilitated review session, gives the risk manager the information needed without compromising the agile component's flexibility. Option A fails because converting agile to predictive just for reporting purposes destroys agile value and is disproportionate. Option C fails because observer access to a sprint board provides raw process data without the analytical framing a risk manager needs. Option D fails because a dependency register alone, replacing both the network diagram and sprint board, loses the visual/structural information that each format provides.

**Mindset Lesson:** In hybrid projects, communicate the right artifact for each track — do not force a single format across different delivery modes.

---

**Q9 — Correct Answer: B**

**PMI Justification:** PMBOK — integrated change control; schedule accuracy. The unauthorized reclassification of a discretionary dependency as mandatory is a change to an approved baseline document without change control approval — it must be corrected. The correct dependency classification (discretionary) should be restored through change control, and the PM should then provide the sponsor with an honest assessment of fast-tracking feasibility and risk. Option A fails because supporting an unauthorized baseline change — even one with protective intent — validates bypassing change control, which is a governance failure. Option C fails because leaving the incorrect label in place to avoid conflict with the sponsor perpetuates a misrepresentation in an approved document. Option D fails because while the unauthorized change is a governance issue, the PM can and should initiate the correction through change control directly; escalation to PMO for "investigation" before correcting a scheduling classification is disproportionate and delays the legitimate sponsor discussion.

**Mindset Lesson:** Unauthorized baseline changes must be corrected through change control regardless of intent — then give the sponsor an honest, complete compression analysis.

---

**Q10 — Correct Answers: A and C**

**PMI Justification:** PMBOK — network diagram quality criteria. (A) is correct because a network with no dangling activities (every interior node connected on both sides) is a fundamental quality criterion for a valid schedule network. (C) is correct because all lags must represent documented real waiting time; undocumented lags represent hidden buffer and are a schedule integrity defect. Option B is incorrect because critical path activities can have discretionary dependencies — criticality and mandatory status are independent. Option D is incorrect because converting discretionary to mandatory is intentional misclassification, which is dishonest and eliminates legitimate compression options. Option E describes desirable convergence but is not a universal requirement — some parallel tracks terminate independently at separate milestones before a final end milestone; the key test is logical completeness, which A captures better.

**Mindset Lesson:** A valid network has no dangling activities and no undocumented lags — completeness and transparency are the two non-negotiable quality tests.

## 12. 60-Second Recap Card

- Sequence Activities produces the schedule network diagram using Precedence Diagramming Method (PDM).
- Four relationship types only: FS (most common), FF, SS, SF (rare). No other types exist in PDM.
- Mandatory dependencies cannot be changed; discretionary dependencies are fast-tracking candidates.
- Leads compress schedule (allow successor to start early); lags add waiting time — both must be technically justified.
- Lags are not schedule buffers — undocumented lags are hidden reserve and must be exposed or removed.
- Mandatory + External dependencies (e.g., BSEC certification, Bangladesh Bank approvals) are hardest constraints; schedule must reflect them realistically.
- Every interior activity needs a predecessor and a successor — dangling activities corrupt critical path.
