# PMP Exam Master Brief: Monitor and Control Project Work

---

## 1. Topic Snapshot — Plain English

Monitor and Control Project Work is the Integration Management process that runs continuously throughout the project, tracking, reviewing, and reporting progress against the performance measurement baseline. It sits in the Monitoring and Controlling process group and feeds directly into Perform Integrated Change Control whenever actual performance deviates from plan. The process consumes work performance data from executing processes and converts it into work performance information, which is then compiled into work performance reports for stakeholder communication. It spans all knowledge areas—schedule, cost, scope, quality, resource, risk, procurement, and stakeholder—making it the broadest single process in the framework. On the exam, this process appears in roughly 15–20% of Integration-tagged questions and bleeds heavily into Risk, Communications, and Stakeholder domains. The distinction between work performance data, information, and reports is a favorite trap. **In one sentence:** Monitor and Control Project Work is the continuous watch-loop that turns raw execution data into actionable intelligence and triggers corrective or preventive action before problems compound.

## 2. The PMI Mindset on This Topic

### Predictive vs Agile vs Hybrid Lens

| Lens | What it looks like |
|---|---|
| Predictive | Formal performance measurement against baselines (scope, schedule, cost); variance analysis; EVM metrics; status reports to sponsor and steering committee on fixed cadence |
| Agile | Continuous monitoring via velocity, burndown/burnup charts, daily standups, sprint reviews; transparency replaces formal reporting; team self-monitors with Scrum Master facilitating |
| Hybrid | Formal EVM at phase-gate or milestone level; agile burndowns within sprints; escalation thresholds defined in project management plan; dual-track dashboards for both stakeholder types |

### Top 5 PMI-isms on This Topic

1. **Work performance data is not information — you must analyze it first.** Real-world instinct says "share the numbers." PMI says raw data (actuals from execution) must be processed into work performance information before it means anything to decision-makers.
2. **Corrective action is not the same as a change request — but it often triggers one.** Practitioners often take informal corrective actions. PMI requires all corrective and preventive actions that affect baselines to flow through the change control process.
3. **The PM's first obligation when variance is detected is to analyze root cause, not report it.** Real-world instinct is to call the sponsor immediately. PMI expects analysis and recommended response options before escalation.
4. **Monitoring is proactive, not reactive.** PMI treats monitoring as continuous risk surveillance, not a periodic checkpoint. Waiting for a status meeting to detect variance is too late.
5. **Forecasts (EAC, ETC) must be updated regularly — not just at period-end.** Real-world PMs update forecasts at milestones. PMI expects rolling forecast updates as new work performance information emerges.

### Ethics Angle

- **Responsibility:** Reporting accurate performance data even when it shows unfavorable variance. In Bangladesh Bank or BSEC-regulated projects, suppressing negative variance to protect a relationship with a regulator violates this value.
- **Honesty:** Work performance reports must reflect reality, not be filtered to manage stakeholder perception. Presenting cherry-picked metrics is a PMI ethics violation.
- **Fairness:** Escalation decisions should be based on objective variance thresholds, not on who the impacted stakeholder is.

## 4. Key Terms, Formulas & Acronyms

| Term / Formula | Definition / Calculation | Exam Trap to Avoid |
|---|---|---|
| Work Performance Data | Raw observations from executing — actuals (cost, schedule, defects, percent complete) | Not actionable by itself; never send raw data to stakeholders as a "report" |
| Work Performance Information | Analyzed data: variances, trends, forecasts; output of M&C processes | Trap: confusing WPI with WPD — WPI requires analysis |
| Work Performance Reports | Packaged communication artifacts (dashboards, status reports, memos) sent to stakeholders | Reports are an output of Monitor and Control Project Work, not of executing |
| EV (Earned Value) | Budgeted cost of work performed = % complete × BAC | Trap: using AC instead of EV to measure progress |
| CV (Cost Variance) | EV − AC; negative = over budget | Negative CV is bad; confusing sign convention is the #1 trap |
| SV (Schedule Variance) | EV − PV; negative = behind schedule | SV in dollars, not time; SPI gives ratio form |
| CPI (Cost Performance Index) | EV / AC; <1 = over budget | CPI <1 means you're getting less value per dollar spent |
| SPI (Schedule Performance Index) | EV / PV; <1 = behind schedule | SPI does not tell you calendar days late |
| EAC (Estimate at Completion) | BAC / CPI (most common); or AC + ETC; or AC + (BAC − EV) | Which formula to use depends on whether past performance is expected to continue |
| VAC (Variance at Completion) | BAC − EAC; tells you projected over/under at end | Negative VAC = projected overrun — triggers escalation |
| TCPI (To-Complete Performance Index) | (BAC − EV) / (BAC − AC) to meet BAC; or (BAC − EV) / (EAC − AC) to meet EAC | TCPI > 1 means you must work more efficiently going forward — use this to reality-check EAC |
| Change Request | Formal request to modify baselines, documents, or processes — output of M&C | Corrective and preventive actions become change requests before implementation |

## 5. PM Behavior Cheat Sheet — FIRST / NEXT / BEST / EXCEPT

| Stem Signal | PM Should FIRST Do | Because |
|---|---|---|
| "Actual costs are exceeding the budget" | Analyze root cause and calculate variance (CV, CPI) | PMI requires analysis before action or escalation |
| "Stakeholder asks for project status" | Provide work performance reports from the communications plan | Reports are the formal output; do not give raw data |
| "Variance exceeds threshold in project plan" | Initiate a change request through integrated change control | Threshold-crossing triggers formal process, not informal fix |
| "Sponsor asks if project will finish on time" | Present EAC, ETC, and updated forecasts with basis of estimate | PMI expects quantified, evidence-based forecasts |
| "Team member implements a workaround without approval" | Document the workaround and raise a change request to formalize it | All scope/baseline changes require formal approval |
| "Schedule is slipping due to resource constraint" | Evaluate options (fast-tracking, crashing, resource reallocation) and present recommendations | Options analysis precedes action — do not act unilaterally |
| "Risk response is not working as planned" | Update risk register, reassess residual risk, consider additional response | Monitoring includes validating risk response effectiveness |
| "Stakeholder complains they were not informed of a delay" | Review communications management plan and issue corrective action | Communications failures are a process gap, not a one-off fix |

**Escalation Edge Cases**

| Stem Signal | PM Should FIRST Do | Because |
|---|---|---|
| "Variance is significant but still within approved threshold" | Document and monitor; do not escalate yet | Escalation is triggered by threshold breach, not by variance size alone |
| "PM cannot resolve an issue that is causing schedule delay" | Escalate to sponsor per the escalation path in the project management plan | PMI escalation requires documented process — ad hoc calls to sponsors bypass governance |

## 6. Visual Memory Aid

```
EXECUTE
   |
   v
Work Performance DATA (raw actuals)
   |
   v
[Analysis: variance, trend, forecast]
   |
   v
Work Performance INFORMATION
   |
   +-----> Within threshold? --> YES --> Update project documents, continue monitoring
   |
   +-----> No (threshold breached?)
              |
              v
         Root cause analysis
              |
              v
         Recommend corrective/preventive action
              |
              v
         CHANGE REQUEST --> Perform Integrated Change Control
              |
              v
         Approved? --> YES --> Implement; Update baselines/docs
                    --> NO  --> Document, monitor, re-assess
```

**Mnemonic: D-I-R-A-C**
- **D**ata collected
- **I**nformation produced (analyze)
- **R**eport distributed
- **A**ction identified (corrective/preventive)
- **C**hange requested if threshold breached

**Memory Hook:** "DIRAC keeps your project on track — physics and PMI both require measurement before action."

Every variance starts as raw data. Only after analysis does it become actionable information. Only after threshold breach does it become a change request. Skipping steps is the exam trap.

## 7. Tailoring Considerations

### Predictive Project
- Formal EVM reporting cycles (weekly, biweekly) aligned to project governance calendar
- Variance thresholds defined in cost/schedule management plans; breach triggers formal change request
- Work performance reports formatted per communications management plan and distributed on schedule
- Lessons learned updates at phase gates, not just project close

### Agile / Hybrid
- Burndown and burnup charts replace EVM as primary monitoring tools within sprints
- Daily standup serves as real-time work performance data collection
- Sprint review is the formal monitoring event where stakeholders validate progress
- Hybrid: EVM at the release level; agile metrics at the sprint level; report both to different audiences
- Retrospective outputs may trigger process changes that require formal change requests at program level

### Industry / Regulatory Factors
- **Bangladesh Bank / BSEC projects:** Regulatory reporting requirements impose mandatory variance disclosure thresholds — project manager must align internal M&C reporting to regulatory disclosure timelines
- **Capital markets technology projects:** System uptime metrics, incident counts, and SLA variances are work performance data inputs; CRO or CCO may require separate risk-specific dashboards
- **Banking core system implementations:** Change Advisory Board (CAB) approval is an external governance layer that may be required before corrective actions are implemented, adding lead time to the corrective action loop
- **Compliance-driven projects:** Suppressing negative variance in status reports to regulators is not just a PMI ethics violation — it is a legal and regulatory risk

## 8. Common Wrong Answers & Why PMI Rejects Them

| Tempting Answer | Why Experienced PMs Pick It | Why PMI Marks It Wrong |
|---|---|---|
| "Immediately inform the sponsor of the cost overrun" | Sponsors need to know; transparency feels like best practice | PMI expects analysis and recommended options before escalation — reporting without analysis is noise, not management |
| "Implement a corrective action to bring project back on track" | Taking action is the PM's job; why wait? | Corrective actions that affect baselines must flow through Perform Integrated Change Control — unilateral implementation bypasses governance |
| "Update the baseline to reflect actual performance" | If actuals diverge, updating the plan seems pragmatic | Baselines are changed only through formal change control; arbitrarily updating them hides variance and distorts reporting |
| "Share the work performance data with stakeholders in the next status meeting" | Transparency is a PMI value; sharing data is transparent | Raw data is not information; stakeholders receive work performance reports (analyzed, contextualized), not raw actuals |
| "Focus monitoring only on critical path activities" | Critical path drives schedule, so monitoring it makes sense | PMI requires monitoring across all performance domains — scope, cost, quality, risk, stakeholder; narrowing focus misses systemic issues |
| "Wait until variance exceeds 10% before acting" | Many organizations use a 10% threshold; it sounds reasonable | The correct threshold is whatever is defined in the project management plan — using an assumed threshold is a governance failure |
| "Ask a team member to fix the problem informally" | Efficient; avoids bureaucratic overhead | Informal fixes that affect scope, schedule, or cost are undocumented changes — they must be formalized through change control |

## 11. Exam-Day Trigger Phrases

| Trigger Phrase in Stem | Your Reflex Answer Pattern |
|---|---|
| "work performance data shows..." | Analyze first → produce work performance information → then act |
| "variance exceeds the threshold" | Raise a change request through Perform Integrated Change Control |
| "sponsor asks for current project status" | Provide work performance reports per communications management plan |
| "the project manager needs to forecast" | Use EAC = BAC/CPI (if past performance continues) or recalculate ETC |
| "corrective action is needed" | Document, raise change request, get approval before implementing |
| "team has implemented a workaround" | Raise change request to formalize the workaround; update risk register |
| "FIRST thing to do when variance detected" | Analyze root cause; do not escalate or act without analysis |
| "what should the PM do NEXT" | Follow the process: data → information → report → change request if needed |
| "the project is behind schedule" | Calculate SV and SPI; evaluate options; present to sponsor with recommendations |
| "stakeholder was not informed" | Identify communications plan gap; issue corrective action to plan, not just to the person |
| "risk response is not effective" | Reassess risk, update risk register, consider contingency reserve use |
| "the PM cannot resolve an issue" | Escalate per escalation path in project management plan |
| "within approved thresholds" | Continue monitoring; do NOT escalate — threshold is the trigger, not just variance existence |

## 9. 10 Hardest Scenario-Based Questions

**Q1 — Ethics Dilemma (Predictive)**

A project manager at a Bangladesh Bank-regulated financial technology firm is completing a monthly status report for the steering committee. The project's CPI is 0.84, and the schedule is two weeks behind. The project sponsor privately tells the PM to "present the numbers favorably" because the regulator's quarterly review is next week and a bad report could trigger an audit. The PM knows the project has approved contingency reserves that partially offset the overrun. What should the PM do?

A. Revise the report to highlight the contingency reserves and present the net exposure as within tolerance, framing the narrative to minimize concern  
B. Report actual CPI and SPI figures with variance analysis and note the contingency reserve position honestly in the report  
C. Delay submitting the status report until after the regulator's quarterly review to avoid triggering scrutiny  
D. Consult with the project sponsor and legal counsel to determine what disclosure is required by the regulator before finalizing the report

**Q2 — Agile Ceremony / Engagement**

A Scrum team is three sprints into a 12-sprint project delivering a trading platform upgrade for a capital markets firm. The product owner reviews the sprint burndown at the daily standup and notices the team is consistently finishing 30% of sprint stories incomplete by sprint end. The velocity is also declining sprint over sprint. The Scrum Master wants to address this. What is the BEST action?

A. Increase sprint length from two weeks to three weeks to give the team more time to complete stories  
B. Reduce the sprint backlog commitment by 30% in the next sprint planning session based on historical velocity  
C. Hold a dedicated retrospective focused on the root causes of incomplete stories and declining velocity before adjusting capacity  
D. Escalate to the project sponsor to request additional resources, since the team is clearly under-resourced

**Q3 — FIRST Action, Conflict (Hybrid)**

A hybrid project manager is running a phase-gated infrastructure rollout with agile delivery teams. At the phase 2 gate review, the EVM report shows SPI of 0.78 and CPI of 0.91. The PMO director wants to place the project on probation and reduce the team's sprint autonomy. The delivery teams argue the agile methodology is working fine within sprints and that the EVM metrics are being measured at the wrong level. What should the PM do FIRST?

A. Defend the agile team's position to the PMO director and request that EVM reporting be suspended for sprint-level work  
B. Agree with the PMO director and implement the additional oversight immediately to protect project continuation  
C. Analyze whether the EVM metrics are being applied at the correct level for the hybrid model and bring findings to the gate review  
D. Escalate to the project sponsor to resolve the conflict between the PMO and the delivery teams

**Q4 — Two-Correct, Pick-Best (Predictive)**

A project manager discovers that actual costs are 18% over budget at the midpoint of a large construction project. The project management plan specifies a 10% cost variance threshold for escalation. The PM has already escalated to the sponsor. What are the TWO BEST next steps?

A. Immediately crash the schedule to recover the budget overrun  
B. Perform a root cause analysis on the cost variance and update the cost management plan  
C. Develop corrective action options and submit a change request through the change control process  
D. Update the cost baseline to reflect actual spending and close the variance  
E. Notify all stakeholders simultaneously of the overrun before presenting options

**Q5 — Ordered Sequence (Agile)**

An agile project team is monitoring a sprint where work performance data shows 40% of user stories are blocked due to an unresolved dependency on an external API team. The Product Owner escalates the issue. Place the following actions in the CORRECT order:

A. Update the impediment log and raise the dependency as a risk in the risk register  
B. Scrum Master removes the impediment by engaging the external API team directly  
C. Document the impact on sprint velocity and update the burndown chart

Options:
A. A → B → C → D  
B. B → A → C → D  
C. C → B → A → D  
D. B → C → A → D

**Q7 — Ethics / Honesty Hardline**

A project manager working on a BSEC-regulated securities system implementation discovers that a previously submitted milestone completion report contained an error — the milestone was reported as 100% complete but 15% of the acceptance criteria had not been tested. **Q9 — Escalation Judgment (Hybrid)**

The project management plan does not explicitly define an escalation threshold for security vulnerabilities, only for cost and schedule variances. The sprint is scheduled to end in two days. What should the PM do?

**Q10 — Multiple-Response (Select 3)**

A project manager is conducting the Monitor and Control Project Work process on a large infrastructure program. Work performance data shows SPI of 0.82 and CPI of 0.94 against a BAC of $12 million. The project management plan cost threshold is CPI < 0.90. Which THREE actions are most appropriate at this stage?

A. Update the issue log with the schedule variance

C. Immediately update the cost baseline to reflect the projected EAC  
D. Perform root cause analysis on the schedule variance to identify corrective options

## 10. Answer Key with Full Justification

**Q1 — Correct Answer: B**

**PMI Justification:** PMI Code of Ethics and Professional Conduct — Honesty value: PMs must provide accurate and truthful information. PMBOK 7 Stewardship principle requires honest reporting. Work performance reports must reflect actual performance data. The sponsor's request to "present favorably" is pressure to distort reporting — a PMI ethics violation regardless of regulatory context. Option A is rationalizing a misrepresentation by using contingency narrative to obscure CPI/SPI actuals. Option C (delay) is a deliberate omission — also dishonest. Option D (consult legal) might be appropriate for determining regulatory disclosure requirements but is secondary to the PM's obligation to report accurately.
**Mindset Lesson:** PMI requires accurate reporting even under stakeholder pressure — rationalizing a favorable narrative framing is still a form of dishonesty.

---

**Q2 — Correct Answer: C**

**PMI Justification:** Agile Practice Guide — retrospectives are the primary vehicle for continuous improvement. Adjusting sprint length (A) changes the process before understanding the problem. Reducing commitment (B) is a valid output of a retrospective but should not be done without root cause analysis first — symptom treatment without diagnosis. Escalating for resources (D) is premature — the root cause may be technical debt, unclear acceptance criteria, or story sizing, not under-resourcing.
**Mindset Lesson:** In agile, data anomalies trigger retrospective inspection before structural changes — diagnose before prescribing.

---

**Q3 — Correct Answer: C**

**PMI Justification:** PMBOK 7 — Integration and measurement performance domains. In a hybrid model, EVM metrics must be applied at the correct governance level. Before defending the team or conceding to the PMO, the PM must determine whether the disagreement is factual (metrics applied incorrectly) or perceptual. Option A (defending team) is advocacy without analysis. Option B (conceding) accepts the PMO's framing without verification. Option D (escalating to sponsor) skips the PM's own analytical responsibility.
**Mindset Lesson:** In hybrid conflicts between governance and delivery, the PM's first move is always to verify the facts, not to take sides.

---

**Q4 — Correct Answers: B and C**

**PMI Justification:** PMBOK 7 — Monitoring and Controlling process group. After escalation, the next steps are root cause analysis (understand why) and change request submission (formalize the corrective action). Option A (crash schedule) addresses schedule, not the cost variance, and is a corrective action that itself requires a change request first. Option D (update baseline) requires formal change control — doing it unilaterally is wrong. Option E (notify all stakeholders simultaneously) skips the analysis step; stakeholders should receive informed options, not raw alerts.
**Mindset Lesson:** Variance triggers analysis and change request — not direct action or baseline revision.

---

**Q5 — Correct Answer: B (B → A → C → D)**

**PMI Justification:** Agile Practice Guide — impediment removal is the Scrum Master's primary responsibility and should happen immediately. Before updating logs (A) or charting impact (C), remove the impediment. Logging first (A) delays resolution. Re-planning (D) is a last resort after impediment removal fails.
**Mindset Lesson:** In agile monitoring, impediment removal precedes documentation — act to unblock, then record.

---

**Q6 — Correct Answer: D (Deliverables)**

**PMI Justification:** PMBOK 6/7 — Deliverables are an output of the Direct and Manage Project Work (Executing) process. They are inputs to Validate Scope and Control Quality, not to Monitor and Control Project Work. Work performance data (B), project management plan (A), issue log (C), and agreements (E) are all valid inputs to Monitor and Control Project Work.
**Mindset Lesson:** Deliverables belong in executing and scope validation — they do not feed the monitoring integration process.

---

**Q7 — Correct Answer: C**

**PMI Justification:** PMI Code of Ethics — Honesty and Responsibility. The error exists; the PM is aware. PMI's honesty standard requires disclosure of material inaccuracies regardless of sign-off status. Option A (quiet correction) is incomplete disclosure — it corrects the artifact but not the stakeholder record. Option B (do nothing) violates honesty directly. Option D (consult legal first) inverts the obligation — legal consultation may follow disclosure, but it does not replace the PM's duty to disclose.
**Mindset Lesson:** Discovered inaccuracies must be disclosed proactively — formal sign-off does not extinguish the PM's honesty obligation.

---

**Q8 — Correct Answer: C**

**PMI Justification:** PMBOK 7 Stakeholder and Communications performance domains; Communications Management Plan drives format, cadence, and content by stakeholder group. Option A (single report) fails the diverse information needs of the three groups — one size fits none. Option B (burndown for all) is inappropriate for an executive EVM audience and a compliance officer requiring exception-based reporting. Option D (delegate to Scrum Master) removes PM accountability for stakeholder communications.
**Mindset Lesson:** Monitoring communication artifacts must be tailored to each audience — the communications management plan defines this, not the PM's preference.

---

**Q9 — Correct Answer: C**

**PMI Justification:** PMBOK 7 Stewardship principle and Risk performance domain — a security vulnerability exposing customer data is a high-severity risk that triggers immediate escalation regardless of documented thresholds. The absence of a threshold in the plan does not mean the PM waits — judgment and stewardship apply. Option A (wait for sprint end) delays a critical risk response. Option B (add to backlog) normalizes a security breach as a standard sprint item. Option D (scope before acting) is appropriate for minor issues, not for customer data exposure.
**Mindset Lesson:** When documented thresholds do not cover a critical risk, PM judgment and stewardship require immediate escalation — governance gaps do not create permission to delay.

---

**Q10 — Correct Answers: B, D, E**

**PMI Justification:** CPI of 0.94 is above the 0.90 threshold — no cost change request is triggered yet, but analysis is appropriate. SPI of 0.82 is a significant schedule concern. B (EAC/TCPI calculation) is the correct analytical step. D (root cause on schedule variance) prepares corrective options. E (change request if baseline change needed) is the correct escalation path for corrective actions. Option A (issue log for schedule variance) is not appropriate — schedule variance is not an issue; it is a performance metric. Option C (update baseline) requires formal change control, not a unilateral act. Option F (inform sponsor of definite overrun) is premature before analysis is complete.
**Mindset Lesson:** Threshold triggers change requests; below-threshold variance triggers analysis and monitoring — distinguish the two.

## 12. 60-Second Recap Card

- Monitor and Control Project Work runs continuously — it is not a periodic checkpoint
- Work performance data (raw) → work performance information (analyzed) → work performance reports (communicated) — never skip the analysis step
- Corrective and preventive actions that affect baselines require a formal change request through Perform Integrated Change Control
- Variance detected → root cause analysis first → options development → escalate if threshold breached
- Baselines are changed only through formal change control — never update them unilaterally to hide variance
- EVM key ratios: CPI = EV/AC; SPI = EV/PV; VAC = BAC − EAC; TCPI = (BAC−EV)/(BAC−AC)
- Stakeholders receive work performance reports, not raw data — format per communications management plan
- In agile, monitoring happens continuously through burndowns, velocity, and retrospectives — not at period-end
- Escalation is triggered by threshold breach per the project management plan — not by variance existence alone
- In regulated environments (Bangladesh Bank, BSEC), accurate reporting is both a PMI ethics obligation and a legal requirement
- **When variance is detected: analyze root cause, quantify impact, develop options, raise a change request — never act unilaterally on anything that touches a baseline.**
