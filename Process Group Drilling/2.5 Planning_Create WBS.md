# PMP Exam Master Brief: Create WBS

---

### 1. Topic Snapshot — Plain English

Create WBS is the process of subdividing project deliverables and project work into smaller, more manageable components — producing the Work Breakdown Structure, the WBS Dictionary, and the Scope Baseline. It is the third process in Project Scope Management within the Planning Process Group and is the bridge between the scope statement and detailed planning: no meaningful schedule, cost estimate, or resource assignment can be developed without a properly structured WBS. The key PMI principle is that the WBS is deliverable-oriented, not activity-oriented — each element represents a deliverable or outcome, never a verb-task. The lowest level of the WBS is the work package, which is the level at which cost and schedule can be estimated and controlled; decomposition below the work package produces activities, which belong to the Schedule Management process. Exam weight is moderate-to-high; this process bleeds into Cost Estimation, Schedule Management, Risk Identification, Procurement, and Quality Management because every subsequent planning baseline depends on WBS completeness and structural integrity. The WBS Dictionary provides the detailed narrative for each WBS element — scope, acceptance criteria, responsible party, cost account, and dependencies — making it the operational heart of scope control. The 100% Rule is PMI's most-tested WBS concept: the WBS must capture 100% of the project scope with no work omitted and no work duplicated. **In one sentence:** Create WBS imposes a deliverable-oriented hierarchy on the project scope, producing the foundation upon which all cost, schedule, quality, and risk baselines are built.

### 2. The PMI Mindset on This Topic

| Lens | What it looks like |
|---|---|
| Predictive | Full WBS developed before schedule and cost baselines; scope baseline = scope statement + WBS + WBS Dictionary; changes require integrated change control |
| Agile | WBS equivalent is a hierarchical product backlog organized by epic/feature/story; decomposition is progressive and just-in-time per iteration |
| Hybrid | High-level WBS captures project structure and known deliverables; lower levels decomposed iteratively; rolling wave planning fills in detail as work approaches |

**Top 5 PMI-isms**

1. The WBS is deliverable-oriented, not activity-oriented — real-world PMs often build task lists when asked for a WBS, but PMI requires every WBS element to be a noun (deliverable or outcome), never a verb (task or action), because activities are derived from work packages, not the other way around.
2. The 100% Rule is absolute — real-world PMs leave implied work out of the WBS to avoid visual complexity, but PMI requires 100% of project scope to be captured in the WBS with no omissions and no duplication, because anything outside the WBS is by definition out of scope.
3. The work package is the lowest WBS level for cost and schedule — real-world PMs continue decomposing until they reach task-level activities in the WBS, but PMI draws a hard line: activities are produced from work packages in Define Activities, not from further WBS decomposition.
4. The WBS Dictionary is as important as the WBS itself — real-world PMs treat the WBS as the primary deliverable and the dictionary as optional documentation, but PMI regards the WBS Dictionary as the document that makes scope control possible because it defines what is and is not included in each work package.
5. Rolling wave planning is not incomplete planning — real-world PMs view partially decomposed WBS elements as planning gaps, but PMI endorses rolling wave planning as a legitimate technique for decomposing near-term work in detail while representing future work at higher levels of the WBS.

**Ethics Angle**

- **Responsibility:** The WBS must reflect the full project scope — omitting work from the WBS to make estimates look favorable is a misrepresentation of project reality.
- **Honesty:** When the PM knows that certain WBS elements are at a summary level because the scope is unclear, that uncertainty must be disclosed to stakeholders rather than presented as complete planning.
- **Fairness:** Responsibility assignments in the WBS Dictionary must be based on capability and availability, not relationship or favoritism.
- **Respect:** Team members should participate in WBS decomposition — imposing a WBS without input from those who will do the work produces estimates they do not own.

### 4. Key Terms, Formulas & Acronyms

| Term / Formula | Definition / Calculation | Exam Trap to Avoid |
|---|---|---|
| Work Breakdown Structure (WBS) | Hierarchical decomposition of the total scope of work into smaller components; each level represents a more granular description of project deliverables | Confusing the WBS with a list of activities — WBS elements are deliverables (nouns); activities are derived from work packages in Define Activities |
| Work Package | The lowest level of the WBS at which cost and duration can be estimated and managed; the unit of scope control | Confusing work packages with activities — activities describe how work is done; work packages describe what is delivered |
| WBS Dictionary | Document providing detailed information about each WBS component: scope description, deliverables, acceptance criteria, responsible organization, resource requirements, cost estimates, quality requirements, technical references | Treating the WBS Dictionary as optional documentation — it is a required component of the scope baseline and the primary scope control reference |
| Scope Baseline | The approved version of the scope statement + WBS + WBS Dictionary; the reference for scope change control | Thinking only the WBS is the scope baseline — all three components (scope statement, WBS, WBS Dictionary) comprise the scope baseline |
| 100% Rule | WBS must capture 100% of the project scope — no work omitted, no work duplicated at any level; each parent level = sum of its children | Thinking the 100% Rule applies only to the lowest level — it applies at every level of the WBS hierarchy |
| Decomposition | Technique of subdividing deliverables into smaller, more manageable components until work packages are reached | Stopping decomposition too early (deliverables not measurable) or too late (going below work package level into activities) |
| Rolling Wave Planning | Technique of elaborating near-term WBS elements in detail while leaving future elements at a higher level; revisited as the project progresses | Treating rolling wave planning as incomplete planning — it is a PMI-endorsed technique, not a planning gap |
| Control Account | Management control point where scope, budget, actual cost, and schedule are integrated and compared; sits above work packages in the WBS | Confusing control accounts with work packages — control accounts aggregate multiple work packages for earned value management |
| Planning Package | WBS component below a control account but above a work package; used when the scope is known but not yet detailed enough to decompose into work packages | Treating planning packages as equivalent to work packages — planning packages require further decomposition before work can be scheduled or costed |
| Code of Accounts | Numbering system used to identify each WBS element uniquely (e.g., 1.2.3.1) | Thinking the code of accounts is only for accounting purposes — it is the traceability backbone linking WBS elements to cost accounts, schedule activities, and risk |
| WBS Component | Any element of the WBS at any level — from the total project at Level 1 down to work packages at the lowest level | Assuming all WBS components are work packages — only the lowest-level deliverable elements are work packages; summary-level elements are WBS components |
| Deliverable-Oriented Hierarchy | The structural principle that every WBS element must represent a deliverable, not a process or activity; the WBS represents "what" not "how" | Building a process-oriented WBS (phases as branches) that lists activities rather than deliverables — phase-based WBS is acceptable only when branches still represent deliverables within each phase |

### 5. PM Behavior Cheat Sheet — FIRST / NEXT / BEST / EXCEPT

| Stem Signal | PM Should FIRST Do | Because |
|---|---|---|
| Team performs work not found in the WBS | Stop the work; assess whether a change request is needed to add it to scope | Work outside the WBS is by definition out of scope; 100% Rule means no work exists outside the WBS |
| WBS element cannot be decomposed because scope is unclear | Apply rolling wave planning; leave at summary level with a planning package until scope clarifies | Rolling wave planning is PMI-endorsed for uncertain future scope; forced decomposition produces fictitious estimates |
| Stakeholder requests a deliverable not in the WBS | Log the request; initiate change request through integrated change control | Post-baseline additions require formal change control regardless of how reasonable the request seems |
| Team estimates are inconsistent across work packages | Review whether work packages are defined at a consistent and manageable level of decomposition | Inconsistent estimates signal work packages at different levels of decomposition; the WBS Dictionary should normalize definition |
| WBS is complete but no dictionary exists | Develop the WBS Dictionary before baselining the scope baseline | The scope baseline requires all three components; a WBS without a dictionary is not a scope baseline |
| PM is asked what is the primary purpose of the WBS | Enable accurate cost and schedule estimation and provide the foundation for scope control | The WBS's primary function is to structure scope for planning and control, not to communicate to executives |
| Work package is too large to estimate reliably | Decompose further until the work package can be reliably estimated within one reporting period | Work packages should be manageable — rule of thumb: estimable in 8–80 hours or within one reporting period |
| Two team members are assigned the same WBS work package | Clarify ownership in the WBS Dictionary; one responsible party per work package prevents accountability gaps | The WBS Dictionary should identify responsible organization for each work package; shared accountability is no accountability |

**Escalation / Weak-Spot Scenarios**

| Stem Signal | PM Should FIRST Do | Because |
|---|---|---|
| Sponsor instructs team to begin work before WBS is baselined | Advise the sponsor of the risk and complete the WBS before work begins; document the sponsor's instruction | Beginning work without a scope baseline means there is no approved reference for scope control — changes cannot be measured against an unestablished baseline |
| Team identifies work necessary for a deliverable that was omitted from the WBS | Assess whether the work is truly within project scope; if yes, add via change request; if no, it is out of scope | Necessary work not in the WBS is still a scope change — necessity does not equal authorization |

### 6. Visual Memory Aid

```
CREATE WBS — Decomposition Flow

START: Project Scope Statement + Requirements Documentation
           |
    [Level 1: Total Project]
           |
    [Level 2: Major Deliverables / Project Phases]
    (e.g., Planning, Design, Build, Test, Deploy)
           |
    [Level 3: Sub-Deliverables under each Level 2]
           |
    ...continue decomposing...
           |
    [Lowest Level: WORK PACKAGES]
           |
    TEST EACH WORK PACKAGE:
    +-------------------------------+
    | Can cost be estimated?  YES/NO |
    | Can duration be estimated? Y/N |
    | Can one owner be assigned? Y/N |
    | Is it within one report period?|
    +-------------------------------+
    ALL YES = valid work package
    ANY NO  = decompose further OR apply rolling wave planning
           |
    [Build WBS Dictionary for each component]
           |
    [Verify 100% Rule: all scope captured, no duplication]
           |
    [Baseline: Scope Statement + WBS + WBS Dictionary = Scope Baseline]
           |
    [Proceed to Define Activities]
```

**Mnemonic:** **D-W-D-100** — Decompose, Work Packages, Dictionary, 100% Rule

**Memory hook:** "Don't Work, Document — 100% or it Doesn't count" — the WBS is useless without the dictionary and the 100% Rule.

**Mnemonic unpacked:**
- **D** — Decompose deliverables until work packages are reached
- **W** — Work packages are the lowest WBS level (not activities)
- **D** — Dictionary describes every WBS element for scope control
- **100** — 100% of scope must appear in the WBS; the Rule is absolute

### 7. Tailoring Considerations

**Predictive Project**
- Full WBS is developed before schedule and cost baselines; the scope baseline must be approved before any execution begins
- WBS Dictionary entries must be thorough — they become the reference for scope control disputes during execution
- Control accounts are positioned in the WBS to enable earned value analysis; their placement must align with reporting requirements
- In fixed-price contracts, the WBS and WBS Dictionary are often contractual documents — imprecise descriptions create claim risk

**Agile / Hybrid**
- In agile, the product backlog organized by epic/feature/story is the WBS equivalent; decomposition occurs at sprint planning through story breakdown
- In hybrid, a summary WBS covers the full project with lower levels progressively elaborated per iteration; rolling wave planning is standard
- User stories in an agile backlog are analogous to work packages — they are the level at which estimation, assignment, and acceptance occur
- Definition of Done for stories serves the same function as the WBS Dictionary entry — it defines what "complete" means for each work package equivalent

**Industry / Regulatory Factors**
- **Banking / Capital Markets / BSEC context:** Regulatory compliance deliverables (audit reports, regulatory filings, penetration test results) must appear as explicit WBS elements — they are verifiable deliverables, not project management activities
- In capital markets infrastructure projects, the WBS should include operational readiness deliverables (runbooks, training, cutover plans) as explicit branches — these are frequently omitted in technical WBS structures, creating late-cycle scope gaps
- BSEC-regulated projects may require the WBS to align with a regulator-specified work breakdown for reporting purposes; the PM must check regulatory guidance before finalizing WBS structure

### 8. Common Wrong Answers & Why PMI Rejects Them

| Tempting Answer | Why Experienced PMs Pick It | Why PMI Marks It Wrong |
|---|---|---|
| Build the WBS as a list of tasks and activities | Experienced PMs equate thorough planning with detailed task lists; they want the team to know what to do | The WBS is deliverable-oriented; activities are derived from work packages in Define Activities, a separate process — mixing activities into the WBS violates the fundamental structure |
| Consider the WBS complete once all work packages are identified; skip the WBS Dictionary | Experienced PMs view the WBS diagram as the scope artifact and treat the dictionary as optional supporting documentation | The WBS Dictionary is a required component of the scope baseline; without it, work packages cannot be controlled because the boundaries of each work package are undefined |
| Decompose the WBS until individual tasks are clearly described | Detail-oriented PMs want unambiguous daily work assignments visible in the WBS | Decomposing below work packages into activities blurs the boundary between WBS and activity list; activities belong in the schedule, not the WBS |
| The scope baseline is the approved WBS alone | Experienced PMs know the WBS is the primary scope planning artifact and assume it constitutes the baseline | The scope baseline consists of three components: scope statement + WBS + WBS Dictionary; none can be omitted |
| Work not explicitly excluded from scope is implicitly in scope | Experienced PMs use logical inference to extend scope to necessary work | PMI applies the inverse: work not in the WBS is out of scope; the 100% Rule requires explicit inclusion, not implicit assumption |
| Apply the same level of decomposition throughout the WBS for consistency | Detail-oriented PMs value uniformity and completeness; inconsistent decomposition looks unprofessional | Rolling wave planning legitimately results in different decomposition depths across the WBS; forcing uniform depth produces fictitious estimates for future uncertain work |
| The WBS should be organized by project phases (Initiate, Plan, Execute, Close) | Experienced PMs use lifecycle phases as WBS Level 2 because projects are managed through phases | Phase-based WBS branches must still contain deliverables at lower levels, not process steps; a WBS organized by phases with activities at the lower levels is activity-oriented, not deliverable-oriented |
| Use a generic WBS template from a previous project without tailoring | Experienced PMs value speed and reuse; similar project types justify template reuse | Generic templates may omit project-specific deliverables or include inapplicable components — the 100% Rule requires every WBS to reflect this project's full scope |

During WBS development, a senior developer tells you that several integration testing deliverables have been excluded from the WBS because "testing is covered by the QA team's plan and doesn't need to be in the WBS." ### 10. Answer Key with Full Justification

**Q1**
**Correct Answer:** B  
**PMI Justification:** The 100% Rule requires 100% of project scope — including testing deliverables — to appear in the WBS. Testing at 15% of project budget is a material deliverable branch, not a supplementary activity. Work outside the WBS is by definition out of scope. Option A delegates the PM's scope definition responsibility. Option C creates a parallel workstream that violates unified scope control. Option D splits the WBS, violating the single integrated scope baseline principle.  
**Mindset Lesson:** The 100% Rule does not recognize "other plans" as a substitute for WBS inclusion — all deliverables belong in the WBS.

**Q2**
**Correct Answer:** B  
**PMI Justification:** Rolling wave planning is the PMI-endorsed approach for WBS elements whose scope cannot yet be fully decomposed. The known portions of the epic should be decomposed to the extent possible; the unclear portions are represented as planning packages. Option A creates fictitious work that misrepresents actual scope. Option C defers legitimate scope work that may still yield value. Option D escalates before applying the available planning technique.  
**Mindset Lesson:** Unknown future scope is handled with rolling wave planning, not placeholder tasks or removal — decompose what you know, plan-package what you don't.

**Q3**
**Correct Answer:** B  
**PMI Justification:** The basellined "Operational Readiness" branch at Level 2 is a summary-level WBS element; adding Level 3 elements changes the scope baseline detail and requires formal change control. Work cannot begin on undefined work packages because there are no WBS Dictionary entries to control against. Option A treats a summary-level WBS element as sufficient authorization for specific work — it is not. Option C creates uncontrolled scope work. Option D removes in-scope work from the project.  
**Mindset Lesson:** A summary-level WBS element is not an authorization to perform any specific work within it — work packages and their dictionary entries must exist before work begins.

**Q4**
**Correct Answer:** B  
**PMI Justification:** "Review architecture design," "Attend daily stand-up," and "Submit status report" are activities (verbs/actions), not deliverables. The WBS must be deliverable-oriented. Activities are derived from work packages in the Define Activities process; they do not belong as WBS elements. Option A misidentifies the issue by labeling it an agile vs. predictive question. Option C misclassifies activities as "management deliverables." Option D misdiagnoses the depth as the problem — the issue is orientation (activity vs. deliverable), not granularity.  
**Mindset Lesson:** If you can describe a WBS element with a verb as its primary word, it is an activity, not a WBS element — restructure until every element is a noun/deliverable.

**Q5**
**Correct Answer:** A (C → B → A → D)  
**PMI Justification:** Scope definition in agile begins with confirming the product vision still bounds the selection (C), then selecting features from the prioritized backlog (B), then decomposing selected features into stories with acceptance criteria (A), then estimating and confirming capacity (D). Vision verification before feature selection prevents selecting features that have drifted outside the product boundary. Estimation comes after decomposition, not before.  
**Mindset Lesson:** Agile decomposition flows from vision to feature selection to story breakdown to estimation — never estimate before you know what you're estimating.

**Q6**
**Correct Answer:** D  
**PMI Justification:** The scope baseline consists of: project scope statement + WBS + WBS Dictionary — three components. The Requirements Traceability Matrix is a project document (not a baseline document) produced in Collect Requirements. It is used throughout the project for traceability but is not part of the scope baseline. Options A, B, and C are all explicitly named scope baseline components per PMBOK.  
**Mindset Lesson:** Scope baseline = three things exactly: scope statement, WBS, WBS Dictionary — the RTM is a project document, not a baseline component.

**Q7**
**Correct Answer:** B  
**PMI Justification:** The 100% Rule requires all project scope — including vendor-managed deliverables that are inputs to in-scope work — to appear in the WBS. A vendor deliverable required for three in-scope items is within the project scope boundary even if the vendor performs it; it must be in the WBS with vendor ownership noted in the WBS Dictionary. Option A incorrectly removes a project-boundary deliverable from scope. Option C documents only the risk without fixing the structural gap. Option D deflects a scope management responsibility to procurement without resolving the WBS omission.  
**Mindset Lesson:** Vendor deliverables that are required inputs to in-scope work are within the project's scope boundary and must appear in the WBS with appropriate ownership documentation.

**Q8**
**Correct Answer:** B  
**PMI Justification:** Before deleting or modifying WBS elements, the PM must verify whether they are truly duplicates or represent distinct deliverables with similar descriptions. If duplication is confirmed, the change must go through integrated change control since the scope baseline is already approved. Option A deletes without verification or change control — both violations. Option C ignores a structural defect in the scope baseline. Option D escalates before completing the PM's own investigation.  
**Mindset Lesson:** Baseline changes — even apparent corrections — require verification and formal change control; do not modify a controlled document without authorization.

**Q9**
**Correct Answer:** B  
**PMI Justification:** A WBS organized around project management phases (Initiate, Plan, Execute, Close) is process-oriented, not deliverable-oriented. The WBS must be structured around the project's deliverables. Project management process phases may appear as one WBS branch (project management deliverables) but should not be the organizing principle for the entire WBS. Option A misidentifies the number of branches as the problem. Option C identifies a valid secondary issue but not the primary structural flaw. Option D is irrelevant to WBS structure.  
**Mindset Lesson:** The WBS organizing principle is deliverables — project management phases are a process tool, not a scope decomposition structure.

**Q10**
**Correct Answers:** A, C, E  
**PMI Justification:**  
- A is correct: WBS Dictionary content per PMBOK explicitly includes scope description, deliverables, acceptance criteria, responsible organization, resource requirements, and quality requirements.  
- B is wrong: The WBS Dictionary is a required component of the scope baseline, not optional documentation.  
- C is correct: During execution, scope control disputes are resolved by reference to the WBS Dictionary entry, which defines what is and is not included in each work package.  
- D is wrong: WBS Dictionary entries are part of the scope baseline; changes to them require integrated change control.  
- E is correct: The scope baseline cannot be approved without all three components; the WBS Dictionary must be complete.  
- F is correct: In agile, the Definition of Done operationally serves the same function as the WBS Dictionary — it defines the conditions under which a work package equivalent (user story) is complete and accepted.  
**Mindset Lesson:** The WBS Dictionary is a baseline control document — it is required, it is part of the scope baseline, and changes to it require formal change control.

### 12. 60-Second Recap Card

- Create WBS decomposes the project scope statement into a deliverable-oriented hierarchy ending at work packages.
- The WBS is deliverable-oriented — every element is a noun (deliverable), never a verb (activity or task).
- The 100% Rule: WBS must capture 100% of project scope — no omissions, no duplication, at every level.
- Work packages are the lowest WBS level where cost and schedule can be estimated; activities are derived from them in Define Activities.
- The scope baseline = scope statement + WBS + WBS Dictionary — all three are required; none can be omitted.
- The WBS Dictionary defines scope, acceptance criteria, responsible organization, and quality requirements for every WBS element.
- Rolling wave planning: decompose near-term work in detail; leave future uncertain work at planning package level — this is PMI-endorsed, not a gap.
- Work outside the WBS is out of scope by definition; adding it after baselining requires integrated change control.
- Control accounts sit above work packages for earned value integration; planning packages sit between control accounts and future work packages.
- Vendor-managed deliverables required for in-scope outputs belong in the WBS with vendor ownership noted in the dictionary.
- **The PM who builds a deliverable-oriented WBS with 100% coverage, a complete dictionary, and rolling wave planning for uncertain scope controls scope better than any PM who starts building before the baseline exists.**
