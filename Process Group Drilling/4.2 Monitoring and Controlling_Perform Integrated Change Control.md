# PMP Exam Master Brief: Perform Integrated Change Control

---

## 1. Topic Snapshot — Plain English

Perform Integrated Change Control is the Integration Management process that reviews all change requests, approves or rejects them, and manages changes to deliverables, project documents, and the project management plan. It is the single gate through which every proposed change to a baseline must pass — no corrective action, no preventive action, no defect repair that touches a baseline is implemented without this process's approval. The Change Control Board (CCB) is the formal decision-making body; the PM may approve minor changes if the CCB delegates that authority, but the authority must be documented. This process runs from project initiation through close and is triggered by outputs from virtually every other process group. On the exam, it accounts for a disproportionate share of Integration and Scope questions and bleeds into Risk, Schedule, Cost, and Quality domains. The key distinction tested is between recommending a change (PM's role) and approving it (CCB's role). **In one sentence:** Perform Integrated Change Control is the governance gate that ensures no baseline changes happen informally — every change is evaluated for impact across all knowledge areas before a decision is made.

## 2. The PMI Mindset on This Topic

### Predictive vs Agile vs Hybrid Lens

| Lens | What it looks like |
|---|---|
| Predictive | Formal CCB reviews change requests; impact assessed against scope/schedule/cost/quality baselines; approved changes update baselines and project documents; rejected changes documented with rationale |
| Agile | Change is expected and welcomed; Product Owner manages scope changes through backlog refinement; sprint scope is locked but product backlog evolves continuously; no formal CCB for intra-sprint changes |
| Hybrid | Formal CCB governs changes at program or phase-gate level; Product Owner manages sprint-level scope changes; a change that crosses sprint into baseline or release plan escalates to CCB |

### Top 5 PMI-isms on This Topic

1. **The PM recommends; the CCB decides.** Real-world instinct is that the PM "owns" the project and approves changes. PMI separates recommendation (PM) from approval authority (CCB or designated authority). Acting on a change before CCB approval is a governance failure.
2. **Every approved change must be reflected in the project management plan and all relevant project documents.** Real-world PMs implement changes and move on. PMI requires that all impacted documents — risk register, schedule, cost baseline, stakeholder register — be updated after approval.
3. **Rejected changes must be documented with rationale — they do not simply disappear.** Real-world instinct is to drop rejected requests. PMI requires they remain in the change log so future decisions have context.
4. **Verbal change approvals are not valid.** In practice, verbal approvals happen constantly. PMI's process requires documented, formal approval before any baseline-impacting change is implemented.
5. **Scope creep is not just unauthorized scope additions — it includes unauthorized schedule or cost changes to accommodate undocumented scope.** Real-world PMs often accommodate small scope additions informally. PMI treats any undocumented baseline change as scope creep.

### Ethics Angle

- **Responsibility:** The PM is responsible for enforcing the change control process even under pressure from powerful stakeholders (e.g., sponsor or regulator directing informal changes).
- **Fairness:** All change requests must be evaluated using the same criteria — stakeholder seniority should not fast-track a change through CCB without proper impact analysis.
- **Honesty:** If a change has been implemented informally before CCB approval, the PM must disclose this and retroactively raise a change request — not conceal the deviation.

## 4. Key Terms, Formulas & Acronyms

| Term / Formula | Definition / Calculation | Exam Trap to Avoid |
|---|---|---|
| Change Request | Formal documented proposal to modify a document, deliverable, or baseline | Verbal requests are not change requests — must be documented |
| CCB (Change Control Board) | Formally chartered group responsible for reviewing and approving/rejecting change requests | PM is not the CCB — PM recommends, CCB approves (unless delegated) |
| Approved Change Request | Change request accepted by the CCB; becomes input to Direct and Manage Project Work for implementation | Approval by PM alone is only valid if the project management plan explicitly delegates authority |
| Rejected Change Request | CCB-denied change; documented in change log with rationale | Rejected changes are not discarded — they are recorded for audit trail and future reference |
| Change Log | Document recording all change requests, their status, and disposition | Separate from issue log; omitting a change from the log is a governance failure |
| Configuration Management System | Processes and tools for tracking and controlling project artifacts and deliverables | Part of PMIS; ensures consistent versioning of documents and deliverables |
| Baseline | Approved plan for scope, schedule, or cost; can only change through formal change control | Never update baselines to match actuals — that erases variance history |
| Scope Creep | Uncontrolled expansion of scope without formal change control | Can be positive (more features) or negative (unauthorized cost/schedule accommodations) |
| Corrective Action | Change request type: realigns future performance to plan | Must go through change control before implementation if it affects a baseline |
| Preventive Action | Change request type: reduces probability of negative future performance | Also requires formal change request and approval before implementation |
| Defect Repair | Change request type: corrects a non-conforming component | Requires formal approval; may require quality control re-inspection after repair |
| PMIS (Project Management Information System) | Software/tools supporting change tracking, versioning, and documentation | Configuration management is a subset of PMIS |

## 5. PM Behavior Cheat Sheet — FIRST / NEXT / BEST / EXCEPT

| Stem Signal | PM Should FIRST Do | Because |
|---|---|---|
| "Stakeholder requests a scope change" | Document the change request formally and submit to CCB for review | Verbal or informal acknowledgment is not a change request — documentation is the first step |
| "Team has already implemented an unapproved change" | Stop the work, document it as an unauthorized change, raise a change request retroactively | Unauthorized changes must be formalized; concealing them is a governance and ethics failure |
| "CCB approves a change request" | Update the project management plan and all relevant project documents | Approval without document update is incomplete — impacted baselines and registers must reflect the change |
| "CCB rejects a change request" | Document the rejection with rationale in the change log | Rejected changes are recorded, not discarded |
| "PM receives verbal approval from sponsor to proceed with a change" | Request written documentation of approval before proceeding | Verbal approvals are not valid in PMI's process — document first |
| "Change request impacts multiple knowledge areas" | Perform integrated impact analysis across scope, schedule, cost, quality, and risk before presenting to CCB | CCB decisions require complete impact analysis — presenting without it leads to uninformed decisions |
| "The project scope is expanding informally" | Identify the scope additions, raise change requests, and enforce the change control process | Informal scope additions are scope creep regardless of stakeholder seniority |
| "A defect is found in a delivered component" | Raise a defect repair change request and submit to CCB | Defect repair is a formal change request type — fixing without approval changes the deliverable baseline |

**Escalation Edge Cases**

| Stem Signal | PM Should FIRST Do | Because |
|---|---|---|
| "Sponsor instructs PM to bypass CCB for an urgent change" | Explain the change control process and request CCB emergency review | Even urgent changes go through CCB — bypass only if project management plan explicitly authorizes the PM to approve |
| "Change is minor and PM believes it is within delegated authority" | Verify the delegated authority threshold in the project management plan before approving | Assuming authority is not the same as having it — verify before acting |

## 7. Tailoring Considerations

### Predictive Project
- CCB composition defined in project charter or project management plan; membership may include sponsor, client, PMO, and technical leads
- Change request form must capture: description, impact on scope/schedule/cost/quality/risk, recommended disposition
- Configuration management system tracks document versions and deliverable baselines
- Change freeze periods near project milestones are common in large predictive programs

### Agile / Hybrid
- Sprint scope is protected during the sprint — changes go to the product backlog, not into the active sprint
- Product Owner is the single authority for backlog prioritization; CCB authority does not override PO for sprint-level decisions
- In hybrid: changes to release baseline or program-level scope escalate to CCB; changes within a sprint managed by PO
- Emergency changes in regulated environments may require CCB expedited review (same-day or 24-hour SLA defined in change management plan)

### Industry / Regulatory Factors
- **Bangladesh Bank / BSEC:** Technology change requests for core banking or securities trading systems may require Bangladesh Bank IT Change Management policy compliance and formal regulatory notification for material changes
- **Capital markets:** Change freeze windows around exchange maintenance, settlement cycles, or regulatory reporting periods — change control must account for blackout calendars
- **Banking core systems:** Change Advisory Board (CAB) in ITIL-governed environments acts as the operational equivalent of the CCB — PM must coordinate project CCB with operational CAB to avoid conflicting approvals
- **Compliance projects:** Any scope reduction that removes a compliance control requires regulatory authority sign-off, not just internal CCB approval

## 8. Common Wrong Answers & Why PMI Rejects Them

| Tempting Answer | Why Experienced PMs Pick It | Why PMI Marks It Wrong |
|---|---|---|
| "Implement the change immediately and document it afterward" | Experienced PMs often fix-first-document-later to maintain momentum | PMI requires approval before implementation for any baseline-impacting change — post-hoc documentation is not the same as pre-approval |
| "Ask the sponsor to approve the change verbally so work can proceed" | Sponsors hold authority; verbal approval feels sufficient in practice | Verbal approvals are not formal approvals in PMI's process — written documentation required before proceeding |
| "Update the project baseline to reflect the approved change" | Updating the baseline after approval sounds like the right next step | The correct action is to update the project management plan AND all relevant project documents — updating only the baseline is incomplete |
| "Reject the change and remove it from the project records" | Rejected changes feel like dead ends not worth preserving | Rejected changes must be documented in the change log with rationale — they are not discarded |
| "The PM approves the change since it is minor" | PMs often have delegated authority for minor changes in practice | PMI requires explicit delegation in the project management plan — assumed authority is not valid authority |
| "Implement the corrective action immediately to bring the project back on track" | Corrective action is urgent; delay seems counterproductive | Corrective actions affecting baselines require a formal change request and CCB approval before implementation |
| "Only submit a change request if the sponsor requests it" | Sponsor direction carries weight; waiting for the sponsor seems deferential | The PM is responsible for initiating change requests when project work requires baseline changes — it is not the sponsor's role to initiate them |

## 9. 10 Hardest Scenario-Based Questions

**Q1 — Ethics Dilemma (Predictive)**

A project manager is overseeing a core banking system upgrade for a bank regulated by Bangladesh Bank. Midway through the project, the sponsor — a C-suite executive — verbally directs the PM to add a new anti-money laundering reporting module to the project scope. The sponsor says the change is strategically critical and that CCB review will "slow things down." The sponsor is also a member of the CCB. What should the PM do?

A. Proceed with the scope addition since the sponsor is a CCB member and his verbal direction constitutes constructive approval  
B. Document the scope addition as a change request and submit it to the full CCB for formal review, informing the sponsor of the process requirement  
C. Add the scope addition to the project plan and begin planning the work, then submit the change request at the next CCB meeting  
D. Escalate the sponsor's request to the project steering committee to avoid conflict with the CCB member

---

**Q2 — Agile Ceremony / Engagement**

A Scrum team is in sprint 4 of a 10-sprint product delivery. During sprint review, a senior business stakeholder says the team should add three new user stories to the current sprint because a competitor has just launched a similar feature. The Product Owner is present. What should the PM do?

A. Instruct the team to add the stories to the current sprint immediately since competitive urgency justifies scope adjustment  
B. Support the Product Owner's authority to add the stories to the product backlog and prioritize them for the next sprint  
C. Raise a formal change request to the CCB for the three new stories since they constitute a scope change  
D. Ask the Scrum Master to facilitate a vote among the development team on whether to accept the stories into the current sprint

**Q3 — FIRST Action, Conflict (Hybrid)**

A hybrid project manager discovers that a team lead has been implementing additional features — not in the approved scope — at the request of a business unit director. The features have been in development for two weeks and are nearly complete. The director says the features are "obviously necessary" and will "clearly be approved." The project schedule has not been impacted yet. What should the PM do FIRST?

A. Allow the team to complete the features since they are nearly done and the schedule is not yet affected  
B. Raise a change request for the unauthorized features and present it to the CCB while stopping further unauthorized work  
C. Document the situation in the issue log and schedule a meeting with the director to discuss change control process  
D. Escalate to the sponsor immediately because the director has bypassed the project manager's authority

**Q4 — Two-Correct, Pick-Best (Predictive)**

A CCB has approved a change request that extends the project schedule by three weeks and increases the cost baseline by $75,000. What are the TWO BEST immediate next steps for the project manager?

A. Notify all stakeholders of the approved change via a status update email  
B. Update the schedule baseline and cost baseline in the project management plan  
C. Update the change log to reflect the approved status and the approved parameters

E. Update the risk register to reflect any new risks introduced by the approved change

**Q5 — Ordered Sequence (Agile)**

A. CCB reviews and approves the change request at an emergency session  
B. Product Owner updates the product backlog and re-prioritizes the release plan  
C. PM performs impact analysis on scope, schedule, cost, and risk of the compliance feature  
D. Team implements the compliance feature in the next sprint

Options:
A. B → C → A → D  
B. C → A → B → D  
C. A → C → B → D  
D. B → A → C → D

**Q7 — Ethics / Honesty Hardline**

A project manager working on a securities trading platform upgrade is aware that a team member implemented a "quick fix" three weeks ago that changed the behavior of a critical calculation module. The fix was never formally submitted as a change request. The system is working correctly, but the baseline documentation is now inaccurate. The PM is preparing for a project audit. What should the PM do?

**Q8 — Communication Strategy**

A project manager has just received CCB approval for a major scope change on a financial system project. The change adds a new customer portal, extends the schedule by six weeks, and increases the budget by $200,000. Seven distinct stakeholder groups are affected. What is the BEST approach to communicating the approved change?

**Q9 — Escalation Judgment (Hybrid)**

## 10. Answer Key with Full Justification

**Q1 — Correct Answer: B**

**PMI Justification:** PMBOK 7 — Integration performance domain; PMI Code of Ethics — Responsibility and Fairness. The change control process applies to all changes regardless of the requester's seniority or CCB membership. A CCB member's verbal direction is not a CCB decision — it is one member's opinion. Submitting formally ensures all CCB members evaluate the impact. Option A rationalizes bypassing process due to role seniority — PMI rejects this. Option C (begin work, then submit) is implementing before approval. Option D (escalate to steering committee) skips the correct process step.
**Mindset Lesson:** CCB membership does not grant individual approval authority — the process protects the project from unilateral decisions, regardless of seniority.

---

**Q2 — Correct Answer: B**

**PMI Justification:** Agile Practice Guide — Product Owner manages the product backlog and sprint scope. Sprint scope is locked; new stories go to the backlog for future sprint prioritization. Option A violates sprint scope protection. Option C misapplies predictive CCB formalism to an agile sprint-level decision — backlog changes are PO authority. Option D bypasses PO authority by involving the team in a scope decision.
**Mindset Lesson:** In agile, competitive urgency drives backlog prioritization — it does not override sprint scope protection or PO authority.

---

**Q3 — Correct Answer: B**

**PMI Justification:** PMBOK 7 — Integration performance domain and change control. Unauthorized work must be stopped and formalized — not completed informally because it is "nearly done." Nearly-done status is irrelevant to change control compliance. Option A normalizes bypassing the process. Option C (document in issue log) is passive — the work must stop. Option D (escalate to sponsor) is premature before the PM attempts to correct the situation through proper process.
**Mindset Lesson:** Unauthorized work does not become authorized by proximity to completion — stop, document, and raise a change request.

---

**Q4 — Correct Answers: B and C**

**PMI Justification:** PMBOK 7 — After CCB approval, the PM must update the project management plan (including schedule and cost baselines) and update the change log with the approved disposition. Option A (notify stakeholders) is important but is downstream of documentation updates. Option D (begin implementing) requires updated plan and work assignments via Direct and Manage Project Work — premature as a "best next step" before plan is updated. Option E (update risk register) is valid but secondary to updating the core plan and log.
**Mindset Lesson:** CCB approval triggers plan updates and change log updates first — communication and implementation follow.

---

**Q5 — Correct Answer: B (C → A → B → D)**

**PMI Justification:** PMBOK 7 — change requests require impact analysis before CCB can make an informed decision. After CCB approval, PO updates the backlog. Implementation happens in the next sprint. Option A (backlog update first) puts PO action before CCB decision. Option C (CCB first without analysis) leads to uninformed approval.
**Mindset Lesson:** Impact analysis precedes CCB review — never put a change to a vote before quantifying what it costs.

---

**Q6 — Correct Answer: D (Work Performance Reports)**

**PMI Justification:** Work performance reports are an output of Monitor and Control Project Work, not of Perform Integrated Change Control. Approved change requests (A), change log updates (B), project management plan updates (C), and project document updates (E) are all valid outputs of Perform Integrated Change Control.
**Mindset Lesson:** Work performance reports belong to the monitoring integration process — change control outputs documentation updates, not performance reporting.

---

**Q7 — Correct Answer: C**

**PMI Justification:** PMI Code of Ethics — Honesty and Responsibility. An unauthorized change exists; the PM is aware. Preparing for an audit without disclosing a known unauthorized change is a form of active concealment. Option A (lessons learned only) does not correct the record. Option B (update documentation quietly) conceals the unauthorized nature of the change. Option D (have team member document it) does not address the disclosure obligation to the CCB and auditors.
**Mindset Lesson:** Retroactive disclosure and change request are required when unauthorized changes are discovered — audits do not create a grace period for concealment.

---

**Q8 — Correct Answer: B**

**PMI Justification:** PMBOK 7 Stakeholder and Communications performance domains; communications management plan governs how, when, and in what format each stakeholder group receives information. Option A (mass email) ignores tailoring requirements. Option C (sponsor cascade) removes PM accountability and risks message distortion. Option D (post and notify) is passive and does not ensure stakeholders receive appropriately tailored information.
**Mindset Lesson:** Approved changes are communicated per the communications management plan — plan updates precede stakeholder notification.

---

**Q9 — Correct Answer: C**

**PMI Justification:** PMBOK 7 Stewardship principle; PMI Code of Ethics — Responsibility. CCB approval is not absolute — it does not override external regulatory requirements. The PM's stewardship obligation requires escalating a compliance risk that the CCB may not have fully understood. Option A (implement because CCB approved) ignores the regulatory context. Option B (risk register and status report) is too slow for a regulatory sanction risk. Option D (new change request) is correct long-term but the immediate step is escalation to senior management and compliance before implementation.
**Mindset Lesson:** CCB approval does not override regulatory compliance obligations — escalate before implementing a change that may violate external law.

---

**Q10 — Correct Answers: B and C**

**PMI Justification:** PMBOK 7 — Rejected change requests are documented in the change log (B) with rationale, and the requestor is formally notified of the decision and rationale (C). Option A (remove from change log) destroys the audit trail — rejected requests are retained. Option D (re-submit immediately) may be appropriate eventually but is not an immediate required action. Option E (update project management plan) is not triggered by a rejection — plan updates follow approvals, not rejections.
**Mindset Lesson:** Rejected changes stay in the log and the requestor is notified — change control creates an audit trail for every decision, not just approvals.

## 12. 60-Second Recap Card

- Every change request must be documented formally before any action is taken — verbal directions are not change requests
- The PM recommends; the CCB (or delegated authority) approves — never assume approval authority
- Impact analysis across all knowledge areas precedes CCB review — scope, schedule, cost, quality, and risk
- CCB approval triggers: update project management plan + update all relevant documents + update change log
- Rejected changes are documented with rationale and retained — they are never discarded
- Unauthorized changes must be stopped and formalized retroactively — proximity to completion does not grant authorization
- Corrective and preventive actions that affect baselines require formal change requests — not informal fixes
- In agile, sprint scope is PO authority; release baseline changes escalate to CCB
- Regulatory compliance requirements override CCB approvals — escalate before implementing changes that may violate external law
- In Bangladesh Bank / BSEC environments, regulatory notification may be required for material scope changes
- **When any change is requested: document it, analyze impact, present to CCB, update the plan on approval — no shortcuts, no verbal approvals, no baseline edits without authorization.**
