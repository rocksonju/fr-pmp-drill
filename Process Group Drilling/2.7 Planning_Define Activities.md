# PMP Exam Master Brief: Define Activities

---

## 1. Topic Snapshot — Plain English

Define Activities is the process of identifying and documenting the specific actions required to produce the project deliverables. It sits within Schedule Management under the Planning Process Group and directly follows the creation of the WBS and WBS dictionary. The output of Define Activities is the activity list, activity attributes, and milestone list — all of which feed directly into Sequence Activities, Estimate Activity Durations, and ultimately Develop Schedule. The process decomposes work packages (WBS lowest level) into schedule activities small enough to be estimated, assigned, monitored, and controlled. In a predictive environment, decomposition is done upfront in planning; in agile/hybrid, activities emerge from backlog refinement and sprint planning. Exam weight for Schedule Management overall is approximately 10–12% of the PMP exam, and Define Activities bleeds into Scope (WBS), Resource Management (assignments), and Communications (activity attributes document owner). **In one sentence:** Define Activities translates WBS work packages into discrete, estimable, assignable schedule activities — the atomic units of project execution.

## 2. The PMI Mindset on This Topic

### Predictive vs Agile vs Hybrid Lens

| Lens | What it looks like |
|---|---|
| Predictive | Activities fully defined upfront from approved WBS; decomposition complete before scheduling begins; activity list baselined with change control |
| Agile | Activities emerge from product backlog refinement; sprint planning defines iteration tasks; no upfront complete activity list required |
| Hybrid | High-level milestones planned predictively; detailed activities for near-term delivery phases defined in rolling wave; regulated deliverables (e.g., BSEC filing, Bangladesh Bank audit report) are fixed; experimental components are agile |

### Top 5 PMI-isms on This Topic

1. **Activities come from work packages, not from scope statements directly.** Real-world PMs often task-list from requirements; PMI requires the WBS → work package → activity decomposition chain to be honored.
2. **Rolling wave planning is legitimate PMI planning, not laziness.** Practitioners instinctively feel they must define all activities before starting; PMI accepts progressive elaboration when future work is genuinely unclear.
3. **Milestones are activities with zero duration — they are not deliverables.** Real-world teams conflate milestones with deliverables; PMI treats milestones as schedule markers requiring explicit documentation in the milestone list.
4. **Activity attributes are not optional metadata.** PMs treat attributes as administrative; PMI uses attributes (predecessor info, resource requirements, imposed dates, constraints, assumptions) as inputs to later schedule processes.
5. **Decomposition stops when the activity can be estimated and assigned.** Real-world instinct pushes either too high (vague tasks) or too low (micro-tasks); the PMI test for "done decomposing" is estimability and assignability, not effort size.

### Ethics Angle

- **Responsibility:** Defining activities completely and honestly, not padding the list to protect schedule float or cutting activities to make a baseline look aggressive.
- **Honesty:** Activity attributes must reflect realistic constraints and assumptions, not what the sponsor wants to see; misrepresenting activity complexity to hit a target date violates PMI honesty norms.
- **Fairness:** In a capital markets project (e.g., BSEC trading system upgrade), defining activities must fairly represent compliance verification steps — omitting regulatory review activities to shorten schedule disadvantages stakeholders relying on that compliance.

## 4. Key Terms, Formulas & Acronyms

| Term / Formula | Definition / Calculation | Exam Trap to Avoid |
|---|---|---|
| Activity List | Comprehensive list of all schedule activities required for the project, including activity identifier and description | Trap: confusing activity list with WBS — WBS shows deliverables/work packages, activity list shows discrete tasks derived from them |
| Activity Attributes | Multiple attributes associated with each activity: predecessors, successors, resource requirements, imposed dates, constraints, assumptions, descriptions | Trap: treating attributes as optional; PMI uses them as inputs to resource, risk, and schedule processes |
| Milestone List | List of significant points or events in the project; milestones have zero duration | Trap: assigning duration to milestones; any milestone with duration is misclassified |
| Work Package | Lowest-level WBS component; deliverable-oriented; decomposed into activities | Trap: scheduling work packages instead of activities — work packages are scope elements, activities are schedule elements |
| Decomposition | Technique of subdividing work packages into smaller, more manageable components (activities) | Trap: stopping decomposition before activities are estimable, or over-decomposing into micro-tasks |
| Rolling Wave Planning | Progressive elaboration technique where near-term work is planned in detail and future work at high level | Trap: treating rolling wave as poor planning; it is a PMI-approved technique for uncertain future work |
| Planning Package | WBS component below control account but above work package; used when scope is not yet decomposed | Trap: scheduling from planning packages — they are placeholders, not schedule-ready work |
| Activity Identifier | Unique alphanumeric code assigned to each activity for tracking and traceability | Trap: omitting identifiers; PMI requires traceability from activity back to WBS work package |
| Precedence Diagramming Method (PDM) | Method using nodes for activities and arrows for dependencies; input from Define Activities to Sequence Activities | Trap: confusing PDM (used in Sequence Activities) with decomposition (used here); Define Activities creates the list, Sequence Activities adds relationships |
| Expert Judgment | Input technique: using SMEs, project team, functional managers to determine completeness of activity list | Trap: relying only on the PM's judgment; team involvement in activity definition is PMI-preferred for accuracy and buy-in |
| Bottom-Up Estimating | Technique where activities are estimated individually and summed; requires a complete activity list as prerequisite | Trap: attempting bottom-up estimates before activity definition is complete — the activity list is a prerequisite |
| Scope Baseline | Approved WBS + WBS dictionary + project scope statement; primary input to Define Activities | Trap: defining activities before scope baseline is approved — activities must trace to approved scope |

## 5. PM Behavior Cheat Sheet — FIRST / NEXT / BEST / EXCEPT

| Stem Signal | PM Should FIRST Do | Because |
|---|---|---|
| "Activities have been defined but team says some are missing" | Review the WBS and scope baseline to identify gaps before adding activities | Activity additions must trace to approved scope; adding without WBS review bypasses traceability |
| "Sponsor wants to start scheduling but WBS is not finalized" | Complete the WBS and get scope baseline approved before defining activities | Activities derive from work packages; scheduling without WBS produces unreliable schedule |
| "Team disagrees on decomposition level of a complex regulatory reporting activity" | Facilitate decomposition workshop using expert judgment from compliance SMEs | Activity definition for regulated deliverables requires domain expertise; PM facilitates, SMEs define |
| "Future project phases are uncertain; stakeholders want a complete activity list now" | Apply rolling wave planning: define near-term activities in detail, document future phases at milestone level | PMI accepts progressive elaboration; forcing premature detail for uncertain work creates false precision |
| "Activity list is complete but PM realizes no milestones are documented" | Create the milestone list, identifying mandatory milestones (contractual, regulatory) and optional milestones | Milestone list is a required output; mandatory milestones (e.g., BSEC submission dates) constrain the schedule |
| "A new regulatory requirement from Bangladesh Bank adds compliance steps mid-planning" | Initiate change request to update scope baseline, then re-examine activity list for new work packages | New regulatory requirements expand scope; activities cannot be added without approved scope change |
| "Team wants to document assumptions about resource availability in the activity list" | Record assumptions in activity attributes, not in the activity list description field | Activity attributes are the correct repository for constraints, assumptions, and resource notes |
| "PM is asked to define activities for a capital markets system implementation with unclear integration scope" | Decompose only the defined scope to work packages and activities; flag undefined integration scope as a risk | Defining activities for undefined scope creates scope creep risk; document the gap and escalate if needed |

**Topic-specific escalation edge cases:**

| Stem Signal | PM Should FIRST Do | Because |
|---|---|---|
| "Sponsor directs PM to omit regulatory audit activities to make the schedule look shorter for board approval" | Decline and document all required activities accurately, then present realistic schedule to sponsor | Omitting required activities violates PMI Honesty and Responsibility; PMI never allows falsifying schedule artifacts |
| "Functional manager says a technical activity is unnecessary; PM disagrees based on risk assessment" | Document the disagreement, retain the activity with supporting rationale in activity attributes, escalate if functional manager insists on removal | PM owns schedule integrity; unilateral removal by functional manager without change control violates governance |

## 6. Visual Memory Aid

```
SCOPE BASELINE (WBS + WBS Dictionary + Scope Statement)
         |
         v
  [Work Packages] — lowest WBS level
         |
    DECOMPOSITION
         |
         v
  [Activity List] — estimable, assignable tasks
  [Activity Attributes] — constraints, assumptions, resource info
  [Milestone List] — zero-duration markers
         |
         v
  --> Sequence Activities (add dependencies)
  --> Estimate Activity Durations (add time)
  --> Estimate Activity Resources (add who/what)
         |
         v
  --> Develop Schedule (integrate all → schedule baseline)

ROLLING WAVE CHECK:
  Near-term work fully defined? --> YES --> Activity List
  Future work uncertain?        --> YES --> Planning Packages + Milestones (refine later)
```

**Mnemonic: WALM** — Work packages → Activity list → List attributes → Milestones

**Memory hook:** "WALM-art stocks activities shelf by shelf" — you walk through the WBS shelf (work packages), pull each item into the activity list, tag it with attributes, and mark the section dividers (milestones).

- **W** = Work packages from WBS (input)
- **A** = Activity list (primary output)
- **L** = aLways document attributes (secondary output)
- **M** = Milestone list (third output)

## 7. Tailoring Considerations

### Predictive Project
- Define all activities before schedule network development; decomposition completeness is a quality gate
- Activity list placed under change control once schedule baseline is approved
- Activity attributes capture imposed dates (contractual deadlines, regulatory submission dates)
- In a multi-year infrastructure project, activities for later phases may be in planning packages pending design completion

### Agile / Hybrid
- Sprint tasks replace a formal activity list; product backlog items decomposed into tasks during sprint planning
- Activity-level definition happens just-in-time, not upfront
- Hybrid: regulatory deliverables (e.g., BSEC quarterly compliance reports, Bangladesh Bank IT audit responses) defined predictively; feature development activities defined iteratively
- Velocity-based capacity planning replaces duration estimation for agile components

### Industry / Regulatory Factors
- **Banking / Capital Markets (BSEC, Bangladesh Bank):** Regulatory submission milestones are mandatory and non-negotiable; they appear in milestone list with imposed dates and are never subject to schedule compression without regulatory approval
- **Capital markets system implementations:** Interface/integration activities with exchange systems (CDBL, DSE) require explicit definition with compliance verification sub-tasks
- **Bangladesh Bank IT guidelines:** Core banking system change activities must include parallel run, UAT sign-off, and go/no-go review as distinct activities — these cannot be merged or omitted
- **Audit-sensitive environments:** Activity traceability to scope baseline is required for internal/external audit; activity identifiers and WBS cross-references are mandatory

## 8. Common Wrong Answers & Why PMI Rejects Them

| Tempting Answer | Why Experienced PMs Pick It | Why PMI Marks It Wrong |
|---|---|---|
| "Start scheduling immediately from the WBS" | Experienced PMs routinely build Gantt charts from WBS deliverables to save time | WBS shows deliverables, not activities; scheduling requires the activity list derived from work packages — skipping decomposition produces an untrackable schedule |
| "Add activities whenever the team identifies new work" | Real-world teams continuously discover work; ad-hoc addition feels responsive | Uncontrolled additions bypass integrated change control; new activities require scope validation before being added |
| "Define activities in as much detail as possible upfront" | Thorough planning feels professional; detail provides control | Over-decomposition for future uncertain work creates false precision and rework when scope changes; rolling wave is PMI-preferred for uncertain phases |
| "Use the project scope statement as the primary input to define activities" | Scope statement describes deliverables; seems like the right starting point | The scope baseline (WBS + WBS dictionary + scope statement) is the input; work packages in the WBS — not the scope statement text — drive decomposition |
| "Assign durations to milestones to mark elapsed time" | Placing time blocks for milestone events helps visualize schedule density | Milestones have zero duration by definition; assigning duration misclassifies them and corrupts schedule calculations |
| "Activity attributes are optional for simple projects" | Small projects seem to need less documentation overhead | PMI treats activity attributes as a required output regardless of project complexity; attributes carry constraint/assumption data needed downstream |
| "PM should define all activities alone to ensure consistency" | A single author produces a consistent list without team debate | Team involvement in activity definition improves accuracy, surfaces hidden work, and increases commitment; PMI consistently favors collaborative planning |
| "Rolling wave planning means the activity list will be incomplete at schedule baseline" | Rolling wave sounds like admitting incompleteness | Rolling wave is a legitimate technique; the activity list is complete for the planned horizon; future phases are documented as planning packages — this is intentional, not incomplete |

## 11. Exam-Day Trigger Phrases

| Trigger Phrase in Stem | Your Reflex Answer Pattern |
|---|---|
| "WBS has been approved" | Define activities is now the correct next step — decompose work packages into activities |
| "Schedule cannot be developed" | Check whether activity list exists; missing activity list is the root blocker |
| "Team wants to start estimating" | Activity list must be complete before estimation; if missing, define activities first |
| "Future phases are unclear" | Apply rolling wave planning; do not force premature full activity definition |
| "Milestone is missing from the schedule" | Update milestone list; milestones must be explicitly documented as zero-duration activities |
| "New regulatory requirement identified" | Submit change request to update scope baseline before adding new activities |
| "Activity list does not trace to WBS" | Traceability is broken; update activity attributes to restore WBS cross-reference |
| "Sponsor wants to omit compliance steps to shorten schedule" | Ethics violation — retain all required activities; escalate if pressure continues |
| "Team disagrees on task granularity" | Facilitate decomposition workshop; apply estimability test — if you can estimate it and assign it, stop decomposing |
| "Planning package is in the schedule" | Planning package is not a schedule activity; it needs further decomposition before scheduling |
| "Activity has been added without a change request" | Integrated change control violation; reverse the addition, submit change request, get approval |

| "Scope creep is occurring during activity definition" | Return to scope baseline; activities must trace to approved work packages |
| "Team identifies work not in WBS during activity definition" | This is undocumented scope; submit scope change request before defining activities for it |

## 9. 10 Hardest Scenario-Based Questions

**Q1 — Ethics Dilemma (Predictive)**

A project manager at a Bangladesh Bank-regulated financial institution is defining activities for a core banking system upgrade. The compliance officer has identified eight mandatory parallel-run verification activities required by Bangladesh Bank IT guidelines. The project sponsor, under pressure from the board to show a compressed schedule for funding approval, instructs the PM to omit these activities from the activity list "for now" and add them back after the funding presentation. The sponsor assures the PM the activities will be reinstated before execution begins. The project manager should:

A. Omit the activities as instructed; the sponsor has authority over schedule presentation decisions and the activities will be added back before execution.
B. Document the activities in a separate internal log but exclude them from the official activity list to satisfy both the sponsor and compliance requirements.
C. Decline to omit the activities, include all required compliance activities in the activity list, and present the accurate schedule to the sponsor with an explanation of the regulatory obligation.
D. Escalate immediately to the PMO before responding to the sponsor, as this is a governance violation requiring formal review before any action.

---

**Q2 — Agile Ceremony / Engagement**

A scrum team is two sprints into a capital markets trading platform implementation. The product owner has written 23 user stories in the backlog but none have been broken into sprint tasks. The development team is struggling to provide sprint commitments during sprint planning because they cannot estimate individual stories without task-level decomposition. The scrum master has raised this as an impediment. What should the team do?

A. Have the product owner convert all 23 user stories into tasks before the next sprint planning session to resolve the estimation problem permanently.
B. Conduct backlog refinement sessions to decompose high-priority stories into tasks; address only the stories planned for the next sprint to keep refinement just-in-time.
C. Skip sprint planning estimation and use yesterday's weather (previous sprint velocity) to commit to a sprint scope based on story points alone.
D. Ask the project manager to define all activities for the entire backlog upfront so the team has a complete task list for estimation.

**Q3 — FIRST Action, Conflict (Hybrid)**

A hybrid project manager is leading a BSEC regulatory reporting system implementation. The project uses a predictive approach for the regulatory compliance deliverables and agile for the internal analytics features. The compliance team has defined a milestone for BSEC quarterly report submission but the development team lead argues that this milestone creates an unrealistic constraint on the agile sprints and wants it removed from the schedule. The PM has just received the activity list for review. What should the PM do FIRST?

A. Remove the BSEC submission milestone to reduce conflict with the agile team and reschedule around the team's capacity.
B. Escalate the disagreement to the project sponsor immediately for a decision on whether to retain the milestone.
C. Explain to the development team lead that BSEC submission milestones are mandatory regulatory constraints that cannot be removed, and work with the team to plan sprint cadence around the fixed milestone dates.
D. Submit a change request to make the BSEC milestone a flexible target date rather than a hard constraint.

**Q4 — Two-Correct, Pick-Best (Predictive)**

A project manager has completed WBS decomposition for a large infrastructure project. Two junior PMs on the team are ready to begin activity definition. One argues that they should define activities for all phases now to have a complete schedule baseline from day one. The other argues that only Phase 1 activities should be fully defined and future phases should use planning packages pending design approval. The project manager must choose an approach. Which response BEST reflects PMI guidance?

A. Define all activities for all phases now; a complete activity list is required before a schedule baseline can be approved.
B. Define Phase 1 activities in detail and document future phases as planning packages; apply rolling wave planning to refine later phases as design is approved.
C. Define high-level activities for all phases now and progressively add detail without using formal planning packages.
D. Wait until all design approvals are complete before beginning any activity definition to avoid rework.

**Q5 — Ordered Sequence (Agile)**

An agile team is about to begin a new sprint. Place the following activity-definition-related steps in the correct order for an agile sprint planning event:

A. Development team estimates task effort; product owner prioritizes stories by value; sprint backlog is finalized; team decomposes selected stories into tasks

**Q7 — Ethics / Honesty Hardline**

A project manager is finalizing the activity list for a capital markets data migration project. A senior developer on the team tells the PM privately that a critical data validation activity has been omitted from the activity list because the functional manager "never approves that kind of rework task." The developer warns that without this activity, data corruption risk is high. The functional manager has already signed off on the current activity list. The PM should:

A. Accept the functional manager's sign-off; the manager has authority over technical decisions and the risk can be tracked in the risk register.
B. Raise the omitted activity through integrated change control; present the data validation activity and its risk justification to the change control board even if it creates conflict with the functional manager.
C. Add the activity to the list informally without submitting a change request, noting it as a "technical sub-task" to avoid a political conflict.
D. Ask the developer to raise the issue directly with the functional manager to keep the PM out of the conflict.

**Q8 — Communication Strategy (Agile or Hybrid)**

A hybrid project team is defining activities for the next delivery increment. The predictive component (regulatory reporting module) has a fixed activity list; the agile component (dashboard features) is being defined sprint by sprint. Stakeholders from the compliance department are frustrated because they cannot see the full activity plan for the agile component. The PM needs to set a communication approach. What is the BEST strategy?

A. Convert the agile component to predictive planning so compliance stakeholders can see a complete activity list.

**Q9 — Escalation Judgment (Hybrid)**

A project manager at a DSE-listed financial services firm is managing a hybrid project with agile sprints for feature development and a predictive track for BSEC compliance deliverables. During activity definition for the compliance track, a legal advisor informs the PM that a new BSEC directive requires two additional activities: a pre-submission legal review and a board-level sign-off meeting. These activities are not in the current approved activity list or scope baseline. The PM's project sponsor says "just add the activities — this is a regulatory requirement, we don't need change control for mandatory items." The PM should:

A. Add the activities as directed by the sponsor; regulatory requirements are inherently approved changes and do not require a formal change request.
B. Initiate a formal change request to update the scope baseline and activity list; regulatory requirements still require integrated change control to update approved plan documents, even when the change is not discretionary.
C. Add the activities to the activity list but do not update the scope baseline; the activity list can be updated independently.
D. Escalate to the PMO before taking any action; the sponsor does not have authority to approve changes to the scope baseline.

**Q10 — Multiple-Response (Select 2)**

A project manager is conducting a quality review of a newly created activity list for a Bangladesh Bank IT compliance project. Which TWO of the following indicate that the activity definition process has been performed correctly?

A. Every activity on the list can be traced to a specific work package in the approved WBS.
B. The activity list includes the project scope statement as an embedded reference document.
C. Each activity is small enough to be estimated by duration and assigned to a resource.
D. Activities have been defined for all project phases including future phases where design is still in progress.
E. The milestone list has been created separately and includes Bangladesh Bank submission milestones with imposed dates.

## 10. Answer Key with Full Justification

**Q1 — Correct Answer: C**

**PMI Justification:** PMI Code of Ethics — Honesty and Responsibility. The PM has a professional obligation to document all required activities accurately regardless of sponsor pressure. Presenting a falsified activity list for funding approval is dishonest and violates the PM's responsibility to stakeholders who rely on the schedule. Bangladesh Bank compliance activities are not discretionary; omitting them exposes the organization and the PM to regulatory and professional risk. Option A fails because sponsor authority does not override professional ethics or regulatory requirements. Option B fails because maintaining a secret internal log while presenting a false official document is still deceptive — it attempts to satisfy two incompatible obligations by hiding the conflict. Option D fails because the PM does not need PMO authorization to uphold basic ethics; the correct action is to decline and explain, not to delay by escalating first.

**Mindset Lesson:** Sponsor authority ends where professional honesty begins — PMI never allows falsifying planning artifacts regardless of organizational pressure.

---

**Q2 — Correct Answer: B**

**PMI Justification:** Agile Practice Guide — backlog refinement as a just-in-time activity definition practice. In agile, tasks are defined immediately before they are needed, not all at once. Decomposing only sprint-planned stories preserves agility and avoids wasteful definition of work that may change. Option A fails because defining all 23 stories' tasks in advance violates just-in-time refinement and creates waste if stories are reprioritized. Option C fails because skipping estimation defeats the purpose of sprint planning — the team cannot make a reliable sprint commitment without task-level visibility. Option D fails because a project manager defining activities for an agile team contradicts the self-organizing team principle; the development team owns sprint task definition.

**Mindset Lesson:** In agile, activity definition is a just-in-time team practice — just enough detail, for just the right stories, at the last responsible moment.

---

**Q3 — Correct Answer: C**

**PMI Justification:** PMBOK 7 — Planning domain; regulatory constraints are imposed external constraints, not negotiable schedule elements. BSEC submission milestones are external regulatory requirements with legal force; they cannot be removed by any internal stakeholder. The PM's first action must be to protect the mandatory constraint and educate the development team, then work collaboratively to adapt sprint planning around it. Option A fails because removing a regulatory milestone violates legal/regulatory compliance obligations. Option B fails because this is not an ambiguous governance decision requiring sponsor resolution — it is a clear regulatory constraint; escalating first wastes time and signals uncertainty. Option D fails because changing a mandatory regulatory date to a flexible target requires regulatory approval, not just an internal change request.

**Mindset Lesson:** In hybrid projects, mandatory regulatory milestones are external constraints — they anchor the schedule, and all other planning adapts to them, not the reverse.

---

**Q4 — Correct Answer: B**

**PMI Justification:** PMBOK 6/7 — Rolling wave planning is an explicitly endorsed technique for progressive elaboration. Forcing activity definition for future phases where design is unapproved creates false precision and rework. Planning packages are the correct placeholder mechanism. Option A fails because PMI does not require a complete activity list for all future phases before approving a schedule baseline; rolling wave is acceptable. Option C fails because high-level activities without formal planning packages provide less control than rolling wave with explicit planning packages. Option D fails because waiting until all design is complete before beginning any activity definition unnecessarily delays planning; near-term activities can and should be defined while design continues.

**Mindset Lesson:** Rolling wave planning is not a planning weakness — it is PMI's endorsed response to genuine future uncertainty.

---

**Q5 — Correct Answer: B**

**PMI Justification:** Agile Practice Guide — Sprint Planning event structure. The correct sequence is: (1) product owner prioritizes by value, (2) team selects stories, (3) team decomposes selected stories into tasks, (4) team estimates task effort, (5) sprint backlog finalized. Value prioritization by the product owner always precedes team selection; task decomposition happens after selection, not before; estimation follows decomposition; sprint backlog formalization is last. Option A fails because it begins with estimation before prioritization and selection. Options C and D present incorrect sequencing of the prioritization/decomposition/estimation chain.

**Mindset Lesson:** In sprint planning, value drives selection, selection drives decomposition, and decomposition enables estimation — never invert this sequence.

---

**Q6 — Correct Answer: D**

**PMI Justification:** PMBOK — Process inputs/outputs. The Work Breakdown Structure (WBS) is an output of Create WBS, not Define Activities. Define Activities takes the scope baseline (which includes the WBS) as an input and produces the activity list, activity attributes, and milestone list. Options A, B, and C are all correct outputs of Define Activities. Option D is the correct EXCEPT answer because the WBS is an input artifact, not an output of this process.

**Mindset Lesson:** The WBS feeds activity definition as an input — Define Activities consumes the WBS, it does not produce it.

---

**Q7 — Correct Answer: B**

**PMI Justification:** PMI Code of Ethics — Responsibility and Honesty. The PM has a professional obligation to surface risks to project outcomes, regardless of political conflict with a functional manager. A missing activity that carries documented data corruption risk must be raised through proper change control channels. The fact that the functional manager already signed off does not eliminate the PM's obligation to correct a known deficiency. Option A fails because functional manager sign-off does not override the PM's obligation when the PM knows the plan is deficient; tracking risk does not substitute for fixing a known process gap. Option C fails because informal additions bypass integrated change control and create accountability gaps. Option D fails because asking the developer to escalate is delegating a PM accountability to a team member; the PM owns schedule and scope integrity.

**Mindset Lesson:** A known deficiency in the activity list cannot be hidden behind a functional manager's signature — the PM owns plan integrity and must act through proper channels.

---

**Q8 — Correct Answer: B**

**PMI Justification:** PMBOK 7 — Stakeholder engagement and tailored communication. The correct approach for hybrid projects is to give stakeholders appropriate visibility into the agile component without forcing it into a predictive format. A roadmap with milestones and regular review touchpoints satisfies transparency needs while preserving agile flexibility. Option A fails because converting agile components to predictive purely to satisfy a documentation preference destroys the value of agile; methodology choice should be driven by work type, not stakeholder comfort. Option C fails because requiring compliance stakeholders to attend every sprint planning session is operationally impractical and provides raw process visibility rather than meaningful communication. Option D fails because creating a detailed activity list for all agile sprints in advance reintroduces waterfall constraints and creates rework when backlog priorities change.

**Mindset Lesson:** In hybrid projects, tailor communication to the delivery mode — give stakeholders roadmap-level visibility into agile components without forcing premature detail.

---

**Q9 — Correct Answer: B**

**PMI Justification:** PMBOK — Integrated change control applies to all changes to approved plan documents, including changes driven by external regulatory requirements. The scope baseline and activity list are approved documents; even mandatory regulatory changes require a formal change request to update them. The sponsor's assertion that regulatory items bypass change control is incorrect — they bypass the decision question (whether to do the work) but not the process of documenting and approving the change to plan documents. Option A fails because "mandatory" does not mean "no change control needed" — it means approval is near-certain but the process still applies. Option C fails because activity list and scope baseline must be updated together; independent activity list updates break scope-schedule traceability. Option D fails because this is not a governance ambiguity requiring PMO intervention; the PM knows the correct action (change control) and should initiate it.

**Mindset Lesson:** Regulatory mandates bypass the approval debate but not the change control process — update plan documents through proper channels even when the change is non-discretionary.

---

**Q10 — Correct Answers: A and C**

**PMI Justification:** PMBOK — Define Activities outputs and quality criteria. (A) is correct because traceability to WBS work packages is the foundational quality check for any activity list — without it, activities are not anchored to approved scope. (C) is correct because estimability and assignability are the PMI tests for whether decomposition is complete — an activity that cannot be estimated or assigned is not yet sufficiently defined. Option B is incorrect because the scope statement is an input, not something embedded in the activity list as a reference. Option D is incorrect because defining activities for future phases where design is still in progress violates rolling wave planning principles — those phases should be in planning packages. Option E describes a correct output (milestone list with imposed dates) but the question tests the activity list quality, and E describes the milestone list, not activity list correctness — more importantly, A and C are the two strongest quality indicators for the activity definition process specifically.

**Mindset Lesson:** The two-point quality test for every activity: can you estimate it, and can you assign it? If yes to both, decomposition is done.

## 12. 60-Second Recap Card

- Define Activities decomposes WBS work packages into discrete, estimable, assignable schedule activities.
- Outputs are: activity list, activity attributes, and milestone list — all three are required.
- Milestones have zero duration; never assign duration to a milestone.
- Activity attributes are not optional metadata — they carry constraints, assumptions, and resource notes used downstream.
- Rolling wave planning is PMI-approved: define near-term activities in detail, use planning packages for uncertain future phases.
- New work discovered during activity definition that is not in the WBS must go through scope change control before activities can be defined for it.
- Regulatory milestones (BSEC, Bangladesh Bank) are mandatory external constraints — they anchor the schedule and cannot be removed by internal stakeholders.
- All changes to the activity list after baseline approval, including mandatory regulatory additions, require integrated change control.
- Sponsor authority does not override the obligation to document all required activities honestly — falsifying the activity list for presentation purposes violates PMI Code of Ethics.
- **PMI reflex: When the activity list is incomplete, nothing downstream (sequencing, estimating, scheduling) is reliable — fix the list first, always.**
