# PMP Exam Master Brief: Control Scope

---

### 1. Topic Snapshot — Plain English

Control Scope is the process of monitoring the status of the project and product scope and managing changes to the scope baseline. It operates in the Monitoring and Controlling Process Group under Scope Management, and its central purpose is to prevent scope creep — the uncontrolled expansion of scope without approved change requests. Every scope change, no matter how minor it appears, must flow through integrated change control before implementation; Control Scope enforces this discipline. Key inputs include work performance data, the scope baseline, the requirements traceability matrix, and the change management plan. Key outputs are work performance information (processed scope variance analysis), change requests, and project document updates. Scope variance analysis compares actual scope delivered against the scope baseline, producing Scope Variance (SV in scope terms) and enabling corrective or preventive action. Cross-domain bleed is significant: Control Scope questions mix with Integrated Change Control, Stakeholder Engagement (scope change requests often originate from stakeholders), and Schedule/Cost Management (scope changes cascade into triple constraint adjustments). The exam weight for Scope Management is approximately 8–10%, and Control Scope questions appear in both predictive and agile contexts.

**In one sentence:** Control Scope enforces that every deviation from the approved scope baseline — regardless of source or apparent size — goes through formal change control before being implemented.

### 2. The PMI Mindset on This Topic

| Lens | What it looks like |
|---|---|
| Predictive | Formal scope baseline (scope statement + WBS + WBS dictionary) governs all work; deviations trigger CRs through CCB; scope variance is calculated and reported |
| Agile | Product backlog serves as the living scope; product owner manages scope through backlog refinement and prioritization; scope changes are welcomed but go through sprint planning, not informal accommodation |
| Hybrid | Phase-level scope baseline governs predictive work; agile increments use backlog management; formal change control applies at the phase boundary; scope creep risks are highest at the interface between agile and predictive components |

**Top 5 PMI-isms**

1. No change is too small to require a change request — real-world PMs accommodate small "quick" changes informally, but PMI treats every scope deviation as requiring formal change control regardless of size.
2. The customer is not always right about scope — experienced PMs often say yes to customer scope requests to maintain relationships; PMI says the PM must evaluate impact and route through CCB before any commitment.
3. Scope creep and scope change are not the same thing — scope creep is uncontrolled change without approval; legitimate scope change is approved and documented; real-world PMs blur this distinction.
4. Work performance information is an output, not an input — PMs often think monitoring generates inputs for decisions; in PMI logic, Control Scope takes raw data (input) and produces analyzed information (output).
5. The scope baseline can only be changed through formal change control — real-world PMs informally update scope documentation; PMI requires CCB approval before the baseline can be revised.

**Ethics Angle**

- Responsibility: Ensuring all scope changes are evaluated for triple constraint impact before approval; protecting the organization from undocumented commitments.
- Honesty: Transparently reporting scope variances and their causes rather than absorbing unauthorized changes into the project quietly.
- Fairness: Applying the change control process consistently to all stakeholders — no informal accommodations for senior sponsors.
- Respect: Acknowledging stakeholder scope requests through the formal process rather than dismissing them or accommodating them without assessment.

### 4. Key Terms, Formulas & Acronyms

| Term / Formula | Definition / Calculation | Exam Trap to Avoid |
|---|---|---|
| Scope Baseline | Approved version of scope statement, WBS, and WBS dictionary | The scope baseline is the ONLY approved measure of scope; informal understanding does not substitute |
| Scope Creep | Uncontrolled expansion of scope without formal approval, budget, or schedule adjustment | Scope creep is not a risk until it happens; it is an active scope management failure |
| Gold Plating | Adding unrequested features or functionality beyond the approved scope | Gold plating violates scope control even when the PM thinks it adds value; PMI always marks it wrong |
| Work Performance Data | Raw observations and measurements from project execution | Input to Control Scope; not yet analyzed — do not confuse with Work Performance Information |
| Work Performance Information | Output of Control Scope: analyzed scope variance data comparing actual to planned | This is the output — processed, contextualized scope status ready for reporting |
| Variance Analysis | Technique: compares actual scope performance to scope baseline to identify variance | Used in Control Scope to determine cause and magnitude of scope deviations |
| Scope Variance (SV - Scope) | Difference between authorized scope and delivered scope | Different from EVM Schedule Variance (SV = EV - PV); context determines which SV the question means |
| Change Request | Output when scope deviation is identified; must go to integrated change control | Corrective action and preventive action are delivered via change requests, not direct team orders |
| Corrective Action | Change request type: action to realign performance with the scope baseline | Do not implement corrective action without a formal CR; field "fixes" are scope creep |
| Requirements Traceability Matrix (RTM) | Links requirements to deliverables and acceptance criteria; updated when scope changes | Input and output of Control Scope; must be updated when approved scope changes occur |
| Configuration Management System | Part of the PMIS; tracks and controls changes to product attributes and baselines | Scope baseline updates must be recorded in the configuration management system |
| Scope Statement | Defines what is and is not in scope; part of the scope baseline | The "what is NOT in scope" section is as important as what is; use it to deflect unauthorized additions |

### 5. PM Behavior Cheat Sheet — FIRST / NEXT / BEST / EXCEPT

| Stem Signal | PM Should FIRST Do | Because |
|---|---|---|
| "Team member adds a feature not in the WBS because the customer mentioned it casually" | Meet with the team member; explain scope control; raise a change request for the additional feature | Informal customer requests do not authorize scope changes; uncontrolled additions are scope creep |
| "Sponsor requests a small change to a deliverable mid-project" | Assess the impact on scope, schedule, and cost; prepare a change request for CCB review | All scope changes require a CR regardless of source or perceived size |
| "Customer is complaining that the delivered product is missing a feature" | Compare the delivered product to the scope baseline and RTM | Before assuming a defect or scope change, confirm whether the feature was ever in the approved scope |
| "Scope variance is identified during status reporting" | Document the variance, analyze root cause, and raise a change request for corrective action | Variance analysis is the technique; the output is a change request, not a direct fix order |
| "PM is asked to approve a team member's informal scope addition after the fact" | Reject retroactive approval; raise a change request; document the unauthorized change | Post-hoc approval of unauthorized work still requires formal change control |
| "New regulatory requirement forces an addition to project scope" | Raise a change request citing the regulatory driver; route through CCB for impact assessment | External drivers do not bypass change control; the CCB decides authorization and baseline updates |
| "Stakeholder insists that a feature was always part of the original scope" | Review the scope baseline, WBS, and RTM with the stakeholder | Documentation governs scope disputes; stakeholder memory is not authoritative |
| "Project is performing below baseline on scope delivery" | Analyze root cause; prepare corrective action change request | Never implement corrective action directly; always via formal CR process |

**Escalation / Weak-Spot Scenarios**

| Stem Signal | PM Should FIRST Do | Because |
|---|---|---|
| "Senior sponsor bypasses CCB and directly instructs team to add scope" | Meet with the sponsor privately; explain the change control process and triple constraint impact | PMI: PM is responsible for enforcing change control even with senior stakeholders |
| "Team is consistently gold plating deliverables across multiple sprints" | Address in retrospective; update the Definition of Done and acceptance criteria to exclude unrequested features | Systematic gold plating is a process gap; address at the process level, not individual level only |

**Mnemonic: D-A-V-C-R**
Data in → Analyze variance → Variance found → Change Request → Route to CCB for Resolution.

**Memory hook:** "DAVID Controls the Room" — D = Data, A = Analyze, V = Variance, I = Investigate root cause, D = Document CR, Controls the Room = CCB decides.

- D = Work performance Data is the input
- A = Analyze variance (actual vs. scope baseline)
- V = Variance identified? Yes → act
- I = Investigate root cause
- D = Document change request
- Controls the Room = CCB has authority, not the PM alone

### 7. Tailoring Considerations

**Predictive Project**
- Scope baseline (scope statement + WBS + WBS dictionary) is the formal control document.
- All scope changes require a written change request with triple constraint impact analysis.
- Configuration management system tracks baseline versions; all approved changes update the baseline version number.
- Large predictive projects (construction, infrastructure) use formal contract change orders as the CR mechanism.

**Agile / Hybrid**
- Backlog refinement is the agile mechanism for scope control; new items are added to the backlog, not implemented immediately.
- Sprint scope is frozen at sprint planning; mid-sprint scope changes require cancellation or deferral to the next sprint, not informal accommodation.
- Hybrid projects must define a clear boundary: which scope is governed by the scope baseline (predictive) and which by the backlog (agile), to prevent scope creep at the interface.
- Product owner backlog prioritization decisions function as the agile equivalent of CCB scope authorization.

**Industry / Regulatory Factors**
- Banking / Capital Markets / BSEC: Regulatory scope changes (new BSEC reporting requirements, Basel III adjustments) must be classified as mandatory changes; CCB still evaluates schedule/cost impact even when authorization is non-negotiable.
- IT / Software: Feature requests from business users are a primary source of scope creep; requirements freeze dates and formal sprint planning gates are critical control points.
- Government / Defense: Contract scope is legally defined; unauthorized scope additions can constitute fraud; configuration management is contractually mandatory.

### 8. Common Wrong Answers & Why PMI Rejects Them

| Tempting Answer | Why Experienced PMs Pick It | Why PMI Marks It Wrong |
|---|---|---|
| "Accommodate small scope changes informally to maintain stakeholder goodwill" | Real-world PMs value relationships and avoid bureaucracy for minor items | PMI: every scope change requires a CR; informal accommodation is scope creep regardless of size |
| "Implement the change and document it afterward to save time" | Post-hoc documentation feels pragmatic and still creates a record | PMI: documentation after unauthorized implementation does not constitute change control; it is scope creep rationalized |
| "Since the customer requested the change, approve it immediately" | Customer requests feel authoritative and PMs want to satisfy sponsors | PMI: customer requests must be evaluated for triple constraint impact and approved by CCB before implementation |
| "Gold plating is acceptable if it adds value without extra cost" | PMs want to delight customers; adding features within budget seems harmless | PMI: gold plating consumes resources, creates unplanned risks, and violates the scope baseline — always wrong |
| "Control Scope only applies to external customer requests" | PMs often see internal team additions as self-managed items | PMI: Control Scope applies to ALL scope deviations regardless of source, including team-initiated additions |
| "Work performance data is the output of Control Scope" | PMs generate data through monitoring activities, so data feels like the product | PMI: work performance data is the INPUT; work performance information (analyzed) is the output |
| "Update the scope baseline directly when an approved change is needed" | Making updates feels efficient; the PM may have CCB authority | PMI: scope baseline updates must go through the configuration management system with version control and formal documentation |
| "If the project is behind on scope delivery, take corrective action immediately" | Experienced PMs are action-oriented and want to fix performance gaps quickly | PMI: corrective action must be submitted as a change request through change control; direct orders to teams bypass governance |

### 9. 10 Hardest Scenario-Based Questions

**Q1**

A project manager is overseeing a financial compliance system implementation for a bank. Midway through the project, the CCO approaches the PM informally during a hallway conversation and says the bank's legal team has identified a new regulatory requirement that must be included in the system. The CCO says "just add it — it's non-negotiable and I have authority to approve it." The PM knows the CCO has organizational authority but is uncertain whether the CCO is an authorized change approver per the change management plan. What should the PM do?

A. Implement the change immediately since the CCO has organizational authority and the requirement is regulatory.
B. Ask the CCO to submit a formal change request so the impact on schedule, cost, and other scope elements can be assessed.
C. Escalate to the project sponsor to confirm whether the CCO's verbal authorization is sufficient.
D. Implement the change and document the CCO's verbal instruction as the authorization record.

**Q2**

An agile team is midway through a two-week sprint when the product owner receives an urgent request from a high-value customer to add a specific reporting feature. The product owner tells the scrum master to "squeeze it in" to the current sprint. The team has capacity to do it. The scrum master is aware the feature was not part of sprint planning. What should the scrum master do FIRST?

A. Add the feature to the current sprint since the product owner has authority and the team has capacity.
B. Tell the product owner the feature can be added to the backlog and prioritized for the next sprint.
C. Raise a change request since the sprint scope is being modified.
D. Consult the team in the daily scrum to get their buy-in before accommodating the request.

**Q3**

A hybrid PM is managing a product launch project. The agile development team completes features via sprints, while the marketing and regulatory review phases are managed predictively with a scope baseline. The product team informally asks the agile developers to add a social media integration feature, arguing it's "just a backend addition." The feature is not in the product backlog or the predictive scope baseline. What should the PM do FIRST?

A. Evaluate whether the feature fits within existing sprint capacity and approve it if it does.
B. Add the feature to the product backlog for the product owner to prioritize in the next sprint planning.
C. Treat the request as a potential scope change; assess impact on both the backlog and the predictive scope baseline; raise a CR for CCB review.
D. Ask the product owner to decide since agile scope is within their authority.

**Q4**

A PM receives a status report showing that the team has delivered 85% of the planned scope deliverables but has consumed 95% of the budget and 90% of the schedule. Two deliverables are partially complete and one has not been started. Scope variance has been identified. Which two actions BEST reflect PMI guidance for this situation?

A. Immediately redirect team resources to the incomplete deliverables to close the scope gap.
B. Conduct variance analysis to determine root cause of the scope performance gap.
C. Raise change requests for corrective action to address the scope and cost variances.
D. Notify the sponsor that the project will miss its deadline.

**Q5**

An agile project team has been receiving informal feature requests from the marketing director through the development team lead, bypassing the product owner entirely. Over three sprints, eight features have been added this way. The product owner discovers this pattern during backlog refinement. Place the following corrective steps in the correct order:

A. Inform the marketing director that all feature requests must go through the product owner.
B. Review the eight features against the original product vision and acceptance criteria.
C. Raise a change request (or backlog item) for each unauthorized feature to formalize their status.
D. Conduct a retrospective to address the process failure that allowed the bypass.

**Q6**

Which of the following is NOT an output of the Control Scope process?

A. Work performance information
B. Change requests
C. Verified deliverables
D. Project document updates

**Q7**

A project manager discovers that a senior developer has been voluntarily adding security enhancements to several modules beyond the agreed scope, believing the additions reduce project risk. The enhancements are technically sound and have not caused delays. The customer has not been informed. What should the PM do?

A. Acknowledge the developer's initiative and allow the enhancements since they reduce risk.
B. Raise a change request for each enhancement; assess scope, schedule, and cost impact; inform the customer.
C. Remove the enhancements from the current deliverables and return to the approved scope.
D. Document the enhancements in lessons learned and allow them to remain since no harm was done.

**Q8**

An agile team is building a mobile banking app for an international bank. The product backlog has 60 stories. Over three PI cycles, the product owner has added 24 new stories based on emerging regulatory requirements and stakeholder feedback without removing any existing stories. The team's velocity has remained constant, meaning the projected delivery date has extended by 40%. The steering committee is unaware of the scope growth. What communication strategy should the PM adopt to address this situation?

A. Continue delivering at current velocity and inform the steering committee only when the full product is ready.
B. Present the steering committee with a scope growth analysis showing the 40-story net increase, timeline impact, and options: reduce scope, extend timeline, or add resources.
C. Ask the product owner to freeze the backlog immediately until the project is back on track.
D. Raise change requests for all 24 new stories retroactively to formalize the additions.

**Q9**

A PM on a hybrid ERP implementation learns that a business unit director has been working directly with the vendor's development team to add custom reports not in the project scope baseline. The vendor has not flagged these additions to the PM. The additions are 60% complete. What should the PM do FIRST?

A. Escalate directly to the steering committee since the director and vendor violated governance.
B. Meet with the vendor and the director separately to understand the full scope of the additions before taking further action.
C. Stop all unauthorized work immediately; document the additions; raise change requests for CCB review.
D. Accept the additions since they are almost complete and removing them would waste effort.

**Q10**

A PM is leading a large infrastructure project. During a quarterly scope review, the following observations are made: (1) Three scope items added without CR approval, (2) Two approved change requests not yet incorporated into the scope baseline, (3) One deliverable completed beyond the approved specification (gold plating), (4) RTM has not been updated in 60 days. Which three corrective actions are MOST appropriate?

A. Raise change requests for the three unauthorized scope additions and submit to CCB.
B. Retroactively approve the unauthorized additions since they are already complete.
C. Update the scope baseline to reflect the two approved but unincorporated CRs.
D. Remove the gold-plated features from the deliverable and return it to approved specification.
E. Update the RTM to reflect current scope status including all approved changes.
F. Issue a warning letter to the team members responsible for the unauthorized additions.

### 10. Answer Key with Full Justification

**Q1**
**Correct Answer:** B
**PMI Justification:** PMBOK 7 — Integrated Change Control; change management plan defines authorized approvers. The CCO's organizational authority does not automatically confer change approval authority; the change management plan governs. Option A bypasses change control. Option C escalates before the simpler step of requesting a formal CR. Option D treats verbal instruction as authorization, which is insufficient.
**Mindset Lesson:** Organizational seniority and change approval authority are separate concepts; always route through the formal change control process regardless of who is asking.

**Q2**
**Correct Answer:** B
**PMI Justification:** Agile Practice Guide — sprint scope is frozen at sprint planning; mid-sprint changes undermine sprint goal integrity. The product owner can add to the backlog and prioritize for the next sprint. Option A accommodates scope creep in an agile context. Option C (formal CR) is a predictive mechanism; in agile, backlog management is the equivalent. Option D delays the correct response.
**Mindset Lesson:** In agile, the sprint scope is a commitment; new requests go to the backlog, not into the current sprint.

**Q3**
**Correct Answer:** C
**PMI Justification:** PMBOK 7 — Hybrid projects require change control at the predictive scope boundary. The feature is not in either the backlog or the scope baseline; it is an unauthorized scope addition. Option A accommodates scope creep. Option B only addresses the agile component. Option D abdicates the PM's responsibility for the predictive scope baseline.
**Mindset Lesson:** In hybrid projects, the PM owns the boundary between agile and predictive scope; changes at the interface require formal change control.

**Q4**
**Correct Answer:** B and C
**PMI Justification:** PMBOK 7 — Monitoring and Controlling; variance analysis leads to change requests for corrective action. Variance analysis determines the root cause (B) before corrective action change requests are raised (C). Option A implements corrective action without a CR. Option D is a communication action but not a primary corrective response.
**Mindset Lesson:** Analyze before acting; root cause analysis informs the right corrective action, which must then be formalized in a change request.

**Q5**
**Correct Answer:** B → C → A → D
**PMI Justification:** PMBOK 7 — Scope Management; unauthorized changes must be assessed, formalized, and process failures corrected. Assess first (B) to understand what was added; formalize (C) to bring items into change control; address the source of future requests (A); and fix the process (D) to prevent recurrence.
**Mindset Lesson:** Unauthorized scope must be assessed and formalized before corrective process actions are taken; never ignore or quietly absorb unauthorized additions.

**Q6**
**Correct Answer:** C
**PMI Justification:** PMBOK 6/7 — Control Scope outputs: work performance information, change requests, project management plan updates, project document updates. Verified deliverables are the output of Control Quality, not Control Scope. Options A, B, D are legitimate Control Scope outputs.
**Mindset Lesson:** Know which process produces verified deliverables (Control Quality) vs. accepted deliverables (Validate Scope) vs. scope status information (Control Scope).

**Q7**
**Correct Answer:** B
**PMI Justification:** PMI Code of Ethics — Responsibility and Honesty; PMBOK scope control. Gold plating violates the scope baseline regardless of technical merit. The customer must be informed; a CR must assess impact. Option A enables gold plating. Option C may be appropriate after assessment but is not the FIRST action. Option D ignores the need for formal assessment and customer notification.
**Mindset Lesson:** Technical merit and good intentions do not authorize scope additions; every unauthorized scope element requires a change request and customer notification.

**Q8**
**Correct Answer:** B
**PMI Justification:** PMBOK 7 — Stakeholder Engagement; transparent reporting of scope growth to decision-makers. The steering committee must have visibility into scope growth and timeline impact to make informed decisions. Option A withholds material information. Option C unilaterally freezes the backlog without governance input. Option D (retroactive CRs) is appropriate but insufficient without stakeholder communication.
**Mindset Lesson:** Scope growth that materially impacts the project timeline must be escalated to the steering committee with options — transparency is a PMI ethical obligation, not a choice.

**Q9**
**Correct Answer:** C
**PMI Justification:** PMBOK 7 — Integrated Change Control; unauthorized work must be stopped, documented, and routed through change control. Option A (immediate escalation) skips the first step of stopping and documenting. Option B (meet separately first) delays stopping the work. Option D accepts completed unauthorized work, which violates change control.
**Mindset Lesson:** Unauthorized work in progress must be stopped first, then documented and routed through change control — sunk cost is never a justification for accepting scope creep.

**Q10**
**Correct Answer:** A, C, E
**PMI Justification:** PMBOK 7 — Scope Control; configuration management; RTM maintenance. A: unauthorized additions require CRs — mandatory. C: approved CRs not yet incorporated into the baseline must be updated — a baseline maintenance obligation. E: RTM must reflect current scope — input and output of Control Scope. Option B: retroactive approval bypasses CCB. Option D: removing gold plating is appropriate but removing features from a delivered item requires a CR, not a unilateral decision. Option F: punitive action is not a primary corrective action in PMI logic.
**Mindset Lesson:** Scope control has three parallel obligations: formalize unauthorized changes, incorporate approved changes into the baseline, and keep the RTM current.

### 12. 60-Second Recap Card

- Control Scope = monitor and manage deviations from the approved scope baseline.
- Scope creep is uncontrolled scope change; even small additions require a formal change request.
- Gold plating is always wrong: adding unrequested features violates scope control regardless of intent or technical merit.
- Key input: work performance data (raw); key output: work performance information (analyzed).
- Variance analysis compares actual vs. scope baseline to identify and quantify scope deviations.
- All scope deviations → Change Request → CCB → decision → update baseline if approved.
- In agile, mid-sprint scope additions go to the backlog; sprint scope is frozen at sprint planning.
- Hybrid projects must define which scope is governed by the baseline vs. the backlog to prevent interface creep.
- The RTM must be updated whenever approved scope changes alter the requirements-to-deliverable linkage.
- Verbal authorization from a senior stakeholder does NOT substitute for a formal change request.
- Corrective action is always delivered through a change request — never a direct order to the team.
- **PMI reflex burn-in: When any scope change appears — regardless of source, size, or urgency — FIRST raise a change request and route it through the CCB.**
