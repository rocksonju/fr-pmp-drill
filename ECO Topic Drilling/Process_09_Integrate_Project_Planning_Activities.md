# PMP Exam Master Brief: Integrate Project Planning Activities (Process · Task 9)

---

## 1. Topic Snapshot — Plain English

Integrating project planning activities is the PM-exclusive discipline of weaving subsidiary plans (scope, schedule, cost, quality, resource, communications, risk, procurement, stakeholder) and baselines (scope, schedule, cost) into one coherent project management plan that reflects all interdependencies. PMI treats integration as the PM's signature accountability — no one else does this. Topics include charter inputs, PM plan development, subsidiary plan coordination, baseline alignment, and progressive elaboration. The Process domain is 50% of the exam; integration appears in many "what does the PM do FIRST" scenarios. In real projects, plans are siloed; on the exam, PMI rewards the option that recognizes interdependencies and resolves conflicts through facilitation, not picking sides.

**In one sentence:** This topic is about integrating subsidiary plans and baselines into one coherent plan — the PM's exclusive accountability.

---

## 2. The PMI Mindset on This Topic

### Predictive vs Agile vs Hybrid Lens

| Lens | Integration Looks Like |
|---|---|
| **Predictive** | Full PM plan with 10 subsidiary plans + 3 baselines; CCB |
| **Agile** | Lightweight charter + release plan + backlog |
| **Hybrid** | Charter + PM plan + iteration planning |

### Top 5 PMI-isms on This Topic

1. **PM owns integration. Period.**
2. **No charter = no project.**
3. **Subsidiary plans must be internally consistent.**
4. **Progressive elaboration is the planning method.**
5. **Tailoring is permitted; absence of integration is not.**

### Ethics Angle

- **Honesty:** Plans reflect reality, not optimism.
- **Responsibility:** Integration cannot be delegated.
- **Respect:** Subsidiary plan owners' input valued.
- **Fairness:** Trade-offs resolved on criteria.

## 4. Key Terms, Formulas & Acronyms

| Term / Formula | Definition / Calculation | Exam Trap to Avoid |
|---|---|---|
| **Project Charter** | Authorization document | Trap: confusing with business case |
| **Project Management Plan** | Integrated subsidiary plans + baselines | Trap: single document |
| **Subsidiary Plans** | Scope, schedule, cost, quality, resource, comms, risk, procurement, stakeholder, change | Trap: forgetting one |
| **Baselines** | Scope, schedule, cost | Trap: changing without ICC |
| **PMB (Performance Measurement Baseline)** | Integrated S/S/C for EVM | Trap: confusing with cost baseline |
| **Configuration Management Plan** | Versioning of artifacts | Trap: confusing with change |
| **Change Management Plan** | Change process | Trap: confusing with configuration |
| **Progressive Elaboration** | Iterative refinement | Trap: confusing with scope creep |
| **Rolling Wave Planning** | Near-term detailed, far-term high-level | Trap: thinking incomplete |
| **Kickoff** | Formal start | Trap: skipping |
| **Project Vision (Agile)** | Lightweight charter equivalent | Trap: skipping |
| **Release Plan** | Multi-sprint planning backbone | Trap: confusing with sprint plan |

## 5. PM Behavior Cheat Sheet — FIRST / NEXT / BEST / EXCEPT

| Stem Signal | PM Should FIRST Do | Because |
|---|---|---|
| **Charter** missing | Pause work; obtain authorization | No charter = no project |
| Two subsidiary plans **conflict** | Facilitate between owners; update conflicting plan via ICC | PM integrates |
| Sponsor asks for **current plan** | Provide approved PM plan + baselines | Source of truth |
| **Subsidiary plan** outdated | Update through ICC | Living document |
| **Agile**: skip release planning | Push back; integration at right altitude | Release plan integrates |
| **New constraint** discovered | Assess across plans; ICC if baseline impact | Integration check |
| PMO asks why **integration plan light** | Show integrated plan + tailoring rationale | Tailoring allowed |
| Closeout requested without **acceptance** | Obtain formal acceptance first | Closure requires validation |
| **PM plan** approved long ago | Reassess; update through ICC | Living |
| **Vendor plan** not integrated | Integrate into subsidiary plans | Integration |

**Escalation-Specific Scenarios:**

| Stem Signal | Correct Escalation Logic |
|---|---|
| Plan conflict cannot be resolved at PM level | Brief sponsor with options and recommendation |
| Sponsor demands scope/schedule/cost commit without integrated analysis | Refuse; do integration; brief honestly |

## 6. Visual Memory Aid

### Integration Flow

```text
CHARTER (authorization)
        │
        ▼
Plan subsidiary processes:
   Scope · Schedule · Cost · Quality · Resource ·
   Communications · Risk · Procurement · Stakeholder · Change
        │
        ▼
Integrate into PM PLAN with three BASELINES (S/S/C)
        │
        ▼
Validate consistency across subsidiary plans
        │
        ▼
Direct + Manage Project Work (executing)
        │
        ▼
Monitor + Control + ICC (governance)
        │
        ▼
Close Project or Phase
```

### Integration Mnemonic: **C-P-D-M-C-C** (PMBOK 6 process names)

> **C**harter → **P**M Plan → **D**irect Work → **M**anage Knowledge → **C**ontrol Work + ICC → **C**lose

Memory hook: **"Charter Plans Direct Managed Controlled Closure."**

## 7. Tailoring Considerations

### Predictive Project

- Full PM plan + 10 subsidiary + 3 baselines + CCB

### Agile / Hybrid

- Project vision + release plan + product backlog
- Subsidiary plans lighter

### Industry / Regulatory Factors

- **Banking / Capital Markets:** documented integration mandatory
- **Public-sector:** charter often political directive

|---|---|---|
| "Implement without charter" | Speed | Unauthorized |
| "Skip ICC for small change" | Speed | All change through ICC |

## 10. Answer Key with Full Justification

---

**Q1 — Correct Answer: B**

**PMI Justification:** ICC non-negotiable.

**Mindset Lesson:** Verbal ≠ ICC.

---

**Q2 — Correct Answer: B**

**PMI Justification:** Release planning integrates cross-team dependencies.

**Mindset Lesson:** Sprint planning ≠ release-level integration.

---

**Q3 — Correct Answer: B**

**PMI Justification:** PM facilitates + ICC update.

**Mindset Lesson:** Subsidiary plan conflicts are PM's to integrate.

---

**Q4 — Correct Answers: A and C**

**PMI Justification:** Charter + plan state reconstruction.

**Mindset Lesson:** Takeover sequence: charter first, plan state second.

---

**Q5 — Correct Answer: A (3 → 4 → 1 → 2)**

**PMI Justification:** Charter → stakeholders → plan → kickoff.

**Mindset Lesson:** Sequence prevents alignment to non-existent target.

---

**Q6 — Correct Answer: E**

**PMI Justification:** Work performance reports are M&C output, not Develop PM Plan input.

**Mindset Lesson:** Process input/output direction matters.

---

**Q7 — Correct Answer: B**

**PMI Justification:** Re-baselining without approved change = hiding variance.

**Mindset Lesson:** Re-baseline = consequence of approved change.

---

**Q8 — Correct Answer: B**

**PMI Justification:** Audience tailoring.

**Mindset Lesson:** Hybrid integration = tailored views, all from single source.

---

**Q9 — Correct Answer: B**

**PMI Justification:** Time pressure accelerates ICC.

**Mindset Lesson:** Urgency does not waive governance.

---

**Q10 — Correct Answers: A, B, C**

**PMI Justification:**

- A: Diagnose why not consulted.
- B: Centralized searchable base.
- C: Kickoff lessons-review.
- D: Longer reports don't equal reuse.
- E: Stopping disproportionate.
- F: KPIs often produce theater.

**Mindset Lesson:** Knowledge reuse = accessibility + ritualized consultation.

## 12. 60-Second Recap Card

**INTEGRATE PROJECT PLANNING ACTIVITIES — Exam Morning Card**

- CPDMCC: Charter · PM Plan · Direct · Manage Knowledge · Control + ICC · Close.
- PM owns integration. Cannot delegate.
- No charter = no project.
- Subsidiary plans + 3 baselines = full PM plan.
- Every change → CR → impact → ICC.
- Re-baseline only after approved change.
- Manage Knowledge continuous.
- Progressive elaboration ≠ scope creep.
- Agile: project vision + release plan + backlog.
- Configuration mgmt = versions. Change mgmt = process.

**Escalation one-liner to burn in:** *PMI escalates change through ICC when impact exceeds PM authority — always with impact assessment and audit trail.*
