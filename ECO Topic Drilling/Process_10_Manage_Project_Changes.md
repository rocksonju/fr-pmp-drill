# PMP Exam Master Brief: Manage Project Changes (Process · Task 10)

---

## 1. Topic Snapshot — Plain English

Managing project changes is the disciplined practice of evaluating every proposed change to baselines, plans, or deliverables through Integrated Change Control (ICC) — assessing impact, routing to the right authority (PM, CCB, or sponsor per change management plan), implementing the approved decision, and updating affected artifacts. PMI treats change discipline as PM hygiene; bypassing ICC for "obvious" or "small" changes is a recurring trap. Topics include change management plan, change request, impact assessment, CCB, change log, configuration management, and ICC. The Process domain is 50% of the exam; change management appears in nearly every "what does the PM do FIRST" scenario. In real projects, change is informal; on the exam, PMI rewards formal change requests + impact assessment.

**In one sentence:** This topic is about routing every change through ICC with impact assessment — no bypass, no verbal approval, no "small" exception.

---

## 2. The PMI Mindset on This Topic

### Predictive vs Agile vs Hybrid Lens

| Lens | Change Management Looks Like |
|---|---|
| **Predictive** | Formal change requests; CCB; change log; ICC |
| **Agile** | In-sprint change discouraged; backlog absorbs change; PO prioritizes |
| **Hybrid** | ICC for baselines + backlog adjustment for delivery scope |

### Top 5 PMI-isms on This Topic

1. **Every baseline change through ICC.**
2. **No verbal approval.**
3. **Impact assessment first.** Then decision.
4. **CCB / sponsor authority above PM threshold.**
5. **Change log captures everything.**

### Ethics Angle

- **Honesty:** Change impacts reported accurately.
- **Responsibility:** ICC discipline maintained.
- **Fairness:** Change requests evaluated on impact, not who submitted.
- **Respect:** Stakeholder change requests heard, evaluated, decided.

## 4. Key Terms, Formulas & Acronyms

| Term / Formula | Definition / Calculation | Exam Trap to Avoid |
|---|---|---|
| **Change Request** | Formal proposal to change baseline/plan/deliverable | Trap: verbal |
| **Change Management Plan** | Process for evaluating changes | Trap: implicit |
| **CCB (Change Control Board)** | Body authorized to approve/reject | Trap: assuming always exists |
| **ICC (Integrated Change Control)** | Process: review + approve + manage | Trap: skipping |
| **Impact Assessment** | Scope/schedule/cost/risk/quality/resource/stakeholder impacts | Trap: scope-only |
| **Change Log** | Living document of all changes | Trap: incomplete |
| **Configuration Management** | Versioning of artifacts | Trap: confusing with change |
| **Baseline Update** | Re-baseline after approved change | Trap: without approval |
| **Workaround** | Reactive response | Trap: confusing with contingency |
| **Versioning** | Configuration mgmt activity | Trap: ad-hoc |

## 5. PM Behavior Cheat Sheet — FIRST / NEXT / BEST / EXCEPT

| Stem Signal | PM Should FIRST Do | Because |
|---|---|---|
| Stakeholder requests change | Submit formal CR; impact assessment; ICC | No bypass |
| Sponsor **verbal approval** | Submit formal CR through ICC | Verbal ≠ written |
| "**Small** change" | Through ICC; impact may surprise | All changes |
| **Multiple CRs** | Each individually traced; no combining for streamlining | Per-change impact |
| Change within **PM authority** | Approve; update baselines + plan + comms | PM-level |
| Change **above PM authority** | Route to CCB / sponsor | Threshold |
| **Defect** found | Issue + maybe CR through ICC if rework affects baseline | Impact assess first |
| Change **affects vendor contract** | Through ICC + procurement coordination | Contract terms |
| Stakeholder **bypasses ICC** | Re-engage; restore process | Discipline |
| CR **rejected** | Log decision; update lessons; notify stakeholders | Decision is decision |

**Escalation-Specific Scenarios:**

| Stem Signal | Correct Escalation Logic |
|---|---|
| CR exceeds PM authority | Route to CCB / sponsor per change mgmt plan |
| Sponsor demands bypass ICC | Refuse; document; escalate to governance if persistent |

## 6. Visual Memory Aid

### ICC Flow

```text
CHANGE NEED
        │
        ▼
Affects baseline / plan / deliverable?
   NO  → Tracked as work performance; no ICC
   YES ↓
        ▼
FORMAL CHANGE REQUEST
        │
        ▼
Impact Assessment (scope, schedule, cost, risk, quality, resources, stakeholders)
        │
        ▼
Within PM authority?
   YES → PM approves; update baselines + PM plan + comms + change log
   NO  ↓
        ▼
Route to CCB / Sponsor per change mgmt plan
        │
        ▼
Approved → update artifacts + change log + lessons + comms
Rejected → log decision + lessons + notify
```

### ICC Mnemonic: **C-R-I-A-U**

> **C**hange Request
> **R**eview impact
> **I**CC (CCB/sponsor decision per plan)
> **A**pprove or reject
> **U**pdate baselines + plan + comms + log

Memory hook: **"CRIA U — every change."**

## 7. Tailoring Considerations

### Predictive Project

- Formal CCB; change log
- Detailed impact assessments

### Agile / Hybrid

- In-sprint change discouraged
- Backlog absorbs scope change
- Cross-team changes through ICC

### Industry / Regulatory Factors

- **Banking / Capital Markets:** audit trail mandatory
- BSEC / Bangladesh Bank: change documentation retained for years
- **Public-sector:** change subject to records retention

## 8. Common Wrong Answers & Why PMI Rejects Them

| Tempting Answer | Why Picked | Why PMI Rejects |
|---|---|---|
| "Implement verbal approval" | Speed | No baseline update or audit |
| "Skip ICC for cosmetic" | Time | ICC determines, not PM |
| "Combine CRs to streamline" | Less paperwork | Hides individual impact |
| "Update only after change implemented" | Pragmatism | Wrong direction |
| "Re-baseline to absorb informal changes" | Clean | Without ICC |
| "Reject without impact analysis" | Discipline | Assessment first |
| "Implement and document later" | Speed | Backward |

## 9. 10 Hardest Scenario-Based Questions

---

**Q1 — Ethics Dilemma (Predictive)**

Sponsor verbally approved scope change. Asks "slip in." 4% contingency, 5-day CP shift. What should you do FIRST?

A. Implement.
B. Formal CR through ICC; impact assessment; present to sponsor.
C. Refuse.
D. Implement with note in steering.

---

**Q2 — Agile Ceremony / Engagement**

Senior stakeholder added 3 stories directly to sprint backlog mid-sprint. Team overcommitted. PO upset. What should you do FIRST?

A. Reject and inform.
B. Engage stakeholder; understand urgency; restore PO ownership; route through PO for prioritization.
C. Escalate to sponsor immediately.
D. Accept and adjust commitments.

---

**Q3 — FIRST Action, Conflict (Hybrid)**

CR submitted by team manager affects vendor contract. Procurement says contract change requires legal. Manager wants decision today. What should you do FIRST?

A. Approve.
B. Engage procurement and legal; impact assessment including contractual; route through ICC per change mgmt plan.
C. Reject.
D. Defer.

---

**Q4 — Two Correct, Pick Best (Predictive)**

Multiple small CRs accumulating. Some related; some independent. Which TWO actions FIRST?

A. Each CR individually impact-assessed.
B. Combine all into single CR to streamline.
C. Route each per change mgmt plan authority.
D. Ignore small ones.
E. Defer all.

---

**Q5 — Ordered Sequence (Predictive)**

CR submitted. Arrange:

1. Update baselines + PM plan + change log + communicate.
2. Route per change mgmt plan (PM/CCB/sponsor).
3. Submit formal CR; impact assessment (scope/schedule/cost/risk).
4. Approved or rejected decision recorded.

A. 1 → 2 → 3 → 4
B. 3 → 2 → 4 → 1
C. 2 → 3 → 1 → 4
D. 4 → 1 → 2 → 3

---

**Q6 — EXCEPT Question (Predictive)**

Which is NOT typically part of impact assessment for a CR?

A. Scope impact
B. Schedule impact
C. Cost impact
D. Risk impact
E. Team morale impact (formal CoQ exclude this)

---

**Q7 — Ethics / Honesty Hardline**

Sponsor instructs you to bypass ICC for a change because "we don't have time for paperwork." Change affects regulatory artifact. What do you do?

A. Bypass.
B. Refuse; explain audit + regulatory exposure; expedite ICC with emergency CCB if available.
C. Bypass and document.
D. Defer.

---

**Q8 — Communication Strategy (Hybrid)**

Multiple CRs in flight. Stakeholders confused about decisions. What is the BEST approach?

A. Stop accepting CRs.
B. Maintain visible change log + periodic CR status update to stakeholders.
C. Hide CRs.
D. Daily detailed CR report to all.

---

**Q9 — Escalation Judgment (Hybrid)**

Vendor CR exceeds PM authority. CCB requires 5 business days. Vendor demands 48-hour decision or withdraws. Impact assessments complete. What is the BEST next step?

A. Approve as PM.
B. Emergency CCB within 48-hour window per change mgmt plan; present assessments; formal decision.
C. Reject.
D. Defer to next CCB.

---

**Q10 — Multiple Response (Agile/Hybrid)**

Stakeholders bypassing ICC repeatedly. Multiple changes implemented informally. Baselines drifting. Which THREE actions MOST appropriate? (Select three.)

A. Investigate why bypass pattern emerged.
B. Reset baselines to current actuals silently.
C. Re-engage stakeholders on ICC discipline; restore process.
D. Brief sponsor with bypass pattern and impact.
E. Mandate stakeholders can't propose changes.
F. Document each bypass as a formal CR retroactively where appropriate.

## 10. Answer Key with Full Justification

---

**Q1 — Correct Answer: B**

**PMI Justification:** Verbal ≠ ICC.

**Mindset Lesson:** Sponsor verbal does not waive ICC.

---

**Q2 — Correct Answer: B**

**PMI Justification:** Restore PO + engage stakeholder.

**Mindset Lesson:** Backlog bypass restored through engagement + ownership.

---

**Q3 — Correct Answer: B**

**PMI Justification:** Multi-party impact assessment + routing.

**Mindset Lesson:** Contract-affecting CRs need cross-functional impact.

---

**Q4 — Correct Answers: A and C**

**PMI Justification:** Individual assessment + proper routing.

**Mindset Lesson:** Each CR per change mgmt plan; no combining for speed.

---

**Q5 — Correct Answer: B (3 → 2 → 4 → 1)**

**PMI Justification:** CR + impact → route → decide → update.

**Mindset Lesson:** Sequence preserves integrity.

---

**Q6 — Correct Answer: E**

**PMI Justification:** Team morale is a downstream consideration, not standard impact dimension.

**Mindset Lesson:** Standard impact: scope/schedule/cost/risk/quality/resources/stakeholders.

---

**Q7 — Correct Answer: B**

**PMI Justification:** Regulatory exposure non-negotiable; emergency CCB.

**Mindset Lesson:** Sponsor urgency does not waive regulatory ICC.

---

**Q8 — Correct Answer: B**

**PMI Justification:** Visible change log + periodic status.

**Mindset Lesson:** Change visibility prevents confusion.

---

**Q9 — Correct Answer: B**

**PMI Justification:** Emergency CCB.

**Mindset Lesson:** Time pressure accelerates ICC.

---

**Q10 — Correct Answers: A, C, D**

**PMI Justification:**

- A: Diagnose pattern.
- C: Restore process.
- D: Brief sponsor.
- B: Re-baselining silently is dishonest.
- E: Banning proposals overreach.
- F: Retroactive CR partial fix; principal is restoration.

**Mindset Lesson:** Bypass pattern = diagnose + restore + brief.

## 12. 60-Second Recap Card

**MANAGE PROJECT CHANGES — Exam Morning Card**

- CRIA U: Change Request · Review impact · ICC · Approve/reject · Update.
- Every baseline change through ICC.
- Verbal approval ≠ ICC.
- No "small" change exception.
- Impact assessment first; decision second.
- CCB / sponsor for above-PM authority.
- Change log captures all.
- Configuration mgmt = versions. Change mgmt = process.
- Each CR per-change; no combining for streamline.
- Re-baseline only after approved change.

**Escalation one-liner to burn in:** *PMI routes every change through ICC with impact assessment — never accepts verbal approval, never bypasses for speed, never combines CRs to streamline.*
