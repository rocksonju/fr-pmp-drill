# PMP Exam Master Brief: Plan and Manage Scope (Process · Task 8)

---

## 1. Topic Snapshot — Plain English

Planning and managing scope is the discipline of defining what the project will deliver (and what it won't), decomposing it into manageable pieces, validating completion with stakeholders, and controlling change. PMI distinguishes product scope (features) from project scope (work), and treats scope baseline (WBS + dictionary + scope statement) as the source of truth. Topics include requirements collection, scope statement, WBS, MoSCoW, Validate Scope (formal acceptance), Control Scope (change), and agile equivalents (product backlog, story map). The Process domain is 50% of the exam; scope threads heavily into integration, change, and quality. In real projects, scope creep is the most common failure; on the exam, PMI rewards disciplined change control and gold-plating refusal.

**In one sentence:** This topic is about defining, decomposing, validating, and controlling what is delivered — through baselines and ICC, not heroic absorption.

---

## 2. The PMI Mindset on This Topic

### Predictive vs Agile vs Hybrid Lens

| Lens | Scope Management Looks Like |
|---|---|
| **Predictive** | Requirements doc; WBS; scope baseline; Control Scope through ICC |
| **Agile** | Product backlog; story map; PO owns backlog; refinement continuous |
| **Hybrid** | Scope baseline at release + agile backlog at iteration |

### Top 5 PMI-isms on This Topic

1. **Scope creep is uncontrolled.** Sanctioned change ≠ creep.
2. **Gold-plating is creep dressed as virtue.** Refuse.
3. **PO owns backlog priority.** Stakeholders propose; PO decides.
4. **Validate Scope ≠ Control Quality.** Acceptance vs correctness.
5. **WBS decomposes 100% of work.** No more, no less.

### Ethics Angle

- **Honesty:** Scope status reported truthfully.
- **Responsibility:** Change control discipline.
- **Fairness:** Backlog prioritization on criteria.
- **Respect:** Stakeholder scope requests heard, evaluated.

## 4. Key Terms, Formulas & Acronyms

| Term / Formula | Definition / Calculation | Exam Trap to Avoid |
|---|---|---|
| **Product Scope** | Features and functions of product | Trap: confusing with project |
| **Project Scope** | Work to deliver product | Trap: confusing with product |
| **Scope Statement** | Document of scope details and exclusions | Trap: skipping exclusions |
| **WBS** | Hierarchical decomposition of project work | Trap: WBS = task list |
| **WBS Dictionary** | Detail per WBS element | Trap: skipping |
| **100% Rule** | WBS = 100% of work, no more no less | Trap: missing or duplicating |
| **Work Package** | Lowest WBS level for control | Trap: too small / too large |
| **Scope Baseline** | Scope statement + WBS + WBS dictionary | Trap: changing without ICC |
| **MoSCoW** | Must / Should / Could / Won't | Trap: all Must |
| **Requirements Traceability Matrix** | Maps requirements to deliverables | Trap: skipping |
| **Validate Scope** | Formal acceptance by stakeholders | Trap: confusing with Control Quality |
| **Control Scope** | Manage scope changes | Trap: bypassing ICC |
| **Scope Creep** | Uncontrolled scope expansion | Trap: confused with sanctioned change |
| **Gold-Plating** | Exceeding requirements | Trap: thinking it's "extra quality" |
| **Product Backlog (Agile)** | Ordered list of features | Trap: PM manages it |
| **Story Map** | Visual product backlog by user journey | Trap: confusing with backlog |
| **DoR / DoD** | Ready vs Done | Trap: confusing |

## 5. PM Behavior Cheat Sheet — FIRST / NEXT / BEST / EXCEPT

| Stem Signal | PM Should FIRST Do | Because |
|---|---|---|
| Stakeholder requests **scope addition** | Through ICC; impact assessment | No bypass |
| **Gold-plating** identified | Refuse; route real change through ICC | Creep |
| Stakeholder **rejects deliverable** | Validate Scope per criteria; ICC if change | Formal acceptance |
| **Scope baseline change** needed | ICC | Authorized only |
| Backlog has **100 items all "Must"** | Re-prioritize with stakeholders | Ruthless prioritization |
| Team **delivered something not in scope** | Surface; check why; align | Scope discipline |
| Sponsor demands **everything now** | Negotiate MVP; defer Should/Could | Value first |
| **WBS** missing | Build it; 100% rule | Baseline foundation |
| **Requirement ambiguous** | Clarify with stakeholder; example | Shared understanding |
| Vendor scope **interpretation differs** | Joint session; single source | Conflicting info |

**Escalation-Specific Scenarios:**

| Stem Signal | Correct Escalation Logic |
|---|---|
| Scope change exceeds PM authority threshold | Through ICC to CCB / sponsor with impact assessment |
| Sponsor demands silent scope reduction (avoid disclosure) | Refuse; document; escalate if persisting |

## 6. Visual Memory Aid

### Scope Management Flow

```text
SCOPE NEED
        │
        ▼
Collect requirements
        │
        ▼
Define scope (statement, exclusions)
        │
        ▼
Create WBS (100% rule)
        │
        ▼
Build WBS dictionary; create RTM
        │
        ▼
Scope baseline (statement + WBS + dictionary)
        │
        ▼
Validate Scope (acceptance) + Control Scope (change)
        │
        ▼
Changes through ICC
```

### Mnemonic: **C-D-W-B-V-C**

> **C**ollect requirements
> **D**efine scope
> **W**BS
> **B**aseline
> **V**alidate Scope
> **C**ontrol Scope

Memory hook: **"Collect, Define, WBS, Baseline, Validate, Control."**

## 7. Tailoring Considerations

### Predictive Project

- Requirements doc; WBS; formal baseline
- Validate Scope at phase gates

### Agile / Hybrid

- Product backlog; story map
- DoR + DoD
- Continuous refinement

### Industry / Regulatory Factors

- **Banking / Capital Markets:** regulatory scope often fixed
- BSEC / Bangladesh Bank: scope changes may require regulator notification
- **Public-sector:** RFP scope; tightly controlled

## 9. 10 Hardest Scenario-Based Questions

---

**Q1 — Ethics Dilemma (Predictive)**

Sponsor verbally approves a scope addition (4% contingency, 5-day critical path shift). Asks you to "just slip it in." Steering in 3 weeks. What should you do FIRST?

A. Implement.
B. Submit formal change request through ICC; impact assessment; present to sponsor before steering.
C. Refuse and inform stakeholder.
D. Implement with note in steering report.

---

**Q2 — Agile Ceremony / Engagement**

Backlog 150 items, all "Must" per various stakeholders. PO overwhelmed. What is the BEST action?

A. Current priority.
B. Facilitate stakeholder prioritization with MoSCoW/WSJF; coach PO on ruthless prioritization; produce real Must list.
C. PO picks top 30 unilaterally.
D. Defer.

---

**Q3 — FIRST Action, Conflict (Hybrid)**

Two stakeholders received conflicting scope info — one from team, one from vendor. Both making decisions on different assumptions. What should you do FIRST?

A. Choose one source.
B. Joint meeting; single source of truth; update communication plan.
C. Escalate to legal.
D. Suspend communication.

---

**Q4 — Two Correct, Pick Best (Predictive)**

You discover team has been silently adding features beyond DoD. Sponsor unaware. Which TWO actions FIRST?

A. Restore DoD compliance.
B. Continue silently.
C. Surface to PO/sponsor; assess scope impact; route through ICC if features have value.
D. Reprimand team.
E. Hide from audit.

---

**Q5 — Ordered Sequence (Predictive)**

Build scope baseline. Arrange:

1. Define scope statement (deliverables, exclusions).
2. Collect requirements from stakeholders.
3. Create WBS (100% rule).
4. Build WBS dictionary.

A. 2 → 1 → 3 → 4
B. 1 → 2 → 3 → 4
C. 3 → 2 → 1 → 4
D. 4 → 3 → 2 → 1

---

**Q6 — EXCEPT Question (Predictive)**

Which is NOT part of the scope baseline?

A. Scope statement
B. WBS
C. WBS dictionary
D. Project charter
E. (Scope baseline includes A, B, C)

---

**Q7 — Ethics / Honesty Hardline**

Sponsor wants you to silently reduce scope to fit timeline without informing stakeholders. The reduced features are visible to users. What do you do?

A. Reduce silently.
B. Refuse; through ICC; communicate scope change to affected stakeholders honestly.
C. Reduce and inform after release.
D. Defer release.

---

**Q8 — Communication Strategy (Hybrid)**

Scope changes are accumulating. Stakeholders confused about current state. What is the BEST approach?

A. Stop accepting changes.
B. Maintain current scope visibility (RTM, backlog state) accessible to stakeholders; communicate change log periodically.
C. Hide changes.
D. Send full backlog daily.

---

**Q9 — Escalation Judgment (Hybrid)**

Scope change request exceeds PM authority. CCB needs to decide. Vendor demands decision in 48 hours or withdraws. Impact assessments complete. What is the BEST next step?

A. Approve to keep vendor.
B. Emergency CCB session with impact assessments; obtain formal decision.
C. Reject.
D. Defer to next CCB; accept vendor withdrawal risk.

---

**Q10 — Multiple Response (Agile/Hybrid)**

Stories are being added to sprints mid-sprint by senior stakeholder bypassing PO. Sprint goal at risk. Which THREE actions MOST appropriate? (Select three.)

A. Investigate bypass pattern; engage stakeholder.
B. Restore PO ownership of backlog.
C. Allow bypass since senior.
D. Surface impact data to sponsor if pattern recurs.
E. Cancel sprint each time.
F. Mandate stakeholders cannot speak to team.

## 10. Answer Key with Full Justification

---

**Q1 — Correct Answer: B**

**PMI Justification:** ICC non-negotiable.

**Mindset Lesson:** Verbal sponsor approval ≠ ICC.

---

**Q2 — Correct Answer: B**

**PMI Justification:** Prioritization is the work.

**Mindset Lesson:** Ruthless prioritization required.

---

**Q3 — Correct Answer: B**

**PMI Justification:** Single source + plan update.

**Mindset Lesson:** Conflicting scope info = joint reconciliation.

---

**Q4 — Correct Answers: A and C**

**PMI Justification:** Restore DoD + surface for legitimate route.

**Mindset Lesson:** Silent feature addition is creep; surface and route.

---

**Q5 — Correct Answer: A (2 → 1 → 3 → 4)**

**PMI Justification:** Requirements → scope statement → WBS → dictionary.

**Mindset Lesson:** Requirements first.

---

**Q6 — Correct Answer: D**

**PMI Justification:** Charter is project authorization, not scope baseline.

**Mindset Lesson:** Scope baseline = statement + WBS + dictionary.

---

**Q7 — Correct Answer: B**

**PMI Justification:** Honesty; ICC; honest communication.

**Mindset Lesson:** Scope reduction visible to users surfaced honestly.

---

**Q8 — Correct Answer: B**

**PMI Justification:** Visibility + periodic change log.

**Mindset Lesson:** Scope state transparent.

---

**Q9 — Correct Answer: B**

**PMI Justification:** Emergency CCB respects governance.

**Mindset Lesson:** Time pressure accelerates ICC, not waives.

---

**Q10 — Correct Answers: A, B, D**

**PMI Justification:**

- A: Engage stakeholder; understand urgency.
- B: Restore PO ownership.
- D: Brief sponsor on pattern if recurs.
- C: Accepts bypass.
- E: Cancel sprint punitive.
- F: Banning speaking is overreach.

**Mindset Lesson:** Stakeholder backlog bypass = engagement + PO ownership restoration + governance brief if pattern.
